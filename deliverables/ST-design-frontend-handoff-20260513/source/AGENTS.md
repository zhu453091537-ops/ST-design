# AGENTS.md

本文件是本项目的长期协作规则。目标是让 Codex 在本仓库内稳定承担“平台建设战略顾问 + 工程执行者”的角色：先理解业务和平台目标，再做方案判断，最后按确认后的范围实施。

## 1. 默认协作流程

1. 先理解需求
   - 复述目标、业务场景、输入输出、边界和约束。
   - 当业务目标、技术边界、页面范围、数据来源、权限要求或验收标准不清楚时，先提问确认。
   - 小范围、意图明确、风险低的修复可以直接执行；涉及架构、全局样式、平台组件、批量修改或不可逆影响时，必须先说明方案并等待确认。

2. 先方案后开发
   - 输出可行性、技术选型、模块划分、样式落点、验证方式和风险。
   - 方案应优先服务平台复用、扩展性、权限边界、维护成本和后续迁移成本。
   - 方案确认后，再进入代码修改、代码示例或分步开发。

3. 分阶段交付
   - 大任务拆成可验证、可回滚的小阶段。
   - 每完成一个阶段或一次代码修改后，输出简短进度报告，说明完成内容、剩余事项、风险和下一步。
   - 最终交付说明必须包含：完成内容、修改文件、验证结果、遗留问题/下一步。

### 1.1 新会话接续摘要规则

为避免长对话自动压缩上下文失败，每轮任务结束时必须输出一份简短的“项目接续摘要”，方便下一次新会话继续开发。

接续摘要只保留下一轮继续开发必需的信息，不要写成长篇报告，不要复制大段代码。

必须包含：

1. 当前目标。
2. 本轮完成内容。
3. 本轮改动文件。
4. 当前未完成问题。
5. 当前阻断点。
6. 关键规则。
7. 禁止修改范围。
8. 下一轮第一优先级。
9. 下一轮不要做什么。
10. 需要验证的页面。
11. 需要执行的检查命令。

要求：

1. 控制在 800 字以内。
2. 结构化输出。
3. 不重复历史背景。
4. 不写无关解释。
5. 每轮结束都输出。
6. 如果任务中断，也要先输出接续摘要再结束。

## 2. 接续文档规则

用户会跨设备、跨聊天接续开发，重要上下文必须沉淀到仓库文档，不能只留在聊天记录里。

### 2.1 开始任务前读取

每次开始新任务、接续旧任务、切换电脑、重开聊天，或用户说“开工继续”时，先读取：

1. `AGENTS.md`
2. `docs/project-log.md`
3. `docs/decision-records.md`
4. `docs/todo-next.md`

读取后先输出“项目接续摘要”，包括：当前项目目标、技术栈、最近完成内容、未完成事项、关键规则、本次建议先做什么。未读取前，不直接开始大规模修改代码。

### 2.2 阶段完成后更新

每次完成阶段性任务后，更新：

1. `docs/project-log.md`：记录日期、任务、完成内容、修改文件、涉及页面/组件、验证结果、遗留问题。
2. `docs/todo-next.md`：记录当前待办、优先级、推荐下一步、暂不处理事项和风险提醒。

典型页面、截图页面、Figma 页面或业务右侧内容页收尾时，必须在收尾报告和 `docs/project-log.md` 中补充“平台治理影响”章节，至少包含：

1. 本轮发现了哪些 ant-design-vue 原生组件问题。
2. 哪些问题已通过平台层解决。
3. 哪些仍是页面临时实现。
4. 哪些页面子组件未来必须回收为平台组件。
5. 后续新页面禁止继续复制哪些实现。
6. 哪些样式应进入主题变量或统一样式入口。
7. 当前仍存在的页面级样式债务。

如果产生长期有效规则、技术决策、组件规范、禁止事项或平台方向，同步更新：

1. `docs/decision-records.md`
2. `AGENTS.md`

### 2.3 接续口令

1. 用户输入“开工继续”时：读取四份文档，输出接续摘要和建议执行顺序，等用户确认后再修改代码。
2. 用户输入“今天结束”时：总结完成和未完成内容，更新 `docs/project-log.md` 与 `docs/todo-next.md`；如有长期规则，再更新 `docs/decision-records.md` 或 `AGENTS.md`。

## 4. 平台组件治理总原则

当需求涉及 Button、Table、Tree、Form、Input、Select、DatePicker、Pagination、Modal、Drawer、Tabs、SearchForm、工具栏、操作列、状态标签、导航、面包屑等通用能力时，默认按平台级问题处理，而不是只改当前页面。

### 4.1 硬性禁止

1. 不修改 `node_modules`。
2. 不修改 ant-design-vue 源码。
3. 不引入 Arco、Element Plus、Naive UI 等其他 UI 组件库替代当前体系。
4. 不为通用组件问题只写当前页面 scoped CSS。
5. 不为了单个页面效果把样式散落到业务页面。
6. 不破坏 ant-design-vue 原有 props、事件、插槽和交互逻辑。
7. 不在业务页面手写顶部导航、左侧菜单或 breadcrumb。

### 4.2 默认落点

通用组件样式和交互优先落到：

1. 平台组件：`apps/web-antd/src/components/platform`
2. 平台组件出口：`apps/web-antd/src/components/platform/index.ts`
3. Vben/Vxe 适配层：如 `#/adapter/vxe-table`
4. 项目全局样式和 token：如 `packages/styles`、设计 token
5. 组件验证页：`apps/web-antd/src/views/platform/typical-page/index.vue`

`/platform/typical-page` 只作为组合验证场，不作为通用视觉样式最终落点。页面 scoped CSS 只允许处理页面布局，如分栏、间距、高度、滚动区域。

平台模块间距默认通过设计 token 维护：模块 / 区块之间默认使用 `--st-layout-section-gap: 24px`，模块内容区默认使用 `--st-module-content-padding: 24px`。页面若确有特殊业务紧凑布局，可以保留局部值，但不得把通用模块间距散落到业务 scoped CSS。

表格工具栏默认由 `PlatformTableToolbar` 统一组织：业务主按钮（如新增、新建，`type="primary"`）必须放在最左侧，业务次要按钮随后；搜索、刷新、设置、全屏等常规工具放在右侧。搜索入口默认表现为右侧工具图标，点击展开输入框，再次点击收起。

标准查询列表页的筛选区默认优先使用 `PlatformQueryPanel`：统一承载查询字段、查询/重置和收起/展开；`PlatformTableToolbar` 只负责表格上方业务按钮与刷新、设置、全屏等通用工具，不再让页面把筛选动作和表格工具混写成两套结构。
`PlatformQueryPanel` 默认以“收起态”展示；除非当前页面在开发前被明确要求默认全部展开，否则一律按收起态进入，点击“展开”后再查看完整筛选项。

`PlatformTable` 表头筛选默认由平台组件接管：筛选面板必须提供“全部”选项，空筛选状态视为默认选中“全部”；选中态和鼠标移入态背景统一使用品牌绿 10% 透明度；筛选项点击即生效，不显示 Antdv 默认的“重置 / 确定”按钮。

`PlatformTable` 默认不根据浏览器高度做表格内部自适应滚动；页码选择展示多少条数据，就应在当前页完整展示多少条，超出浏览器可视高度时由页面整体上下滚动。只有页面在开发时被明确要求“表格内部滚动”或显式传入 `scroll.y` / 开启自适应能力时，才允许回到内部滚动模式。

`PlatformTable` 和 `#/adapter/vxe-table` 默认提供“序号”列；后续新增表格时，如用户没有明确说明不要序号列，必须默认展示序号。确有特殊页面不需要序号时，调用方必须显式关闭并说明原因。

`PlatformStatCard` 统计卡的图标容器尺寸、内部图标尺寸、背景、边框、文本色、阴影等视觉值必须优先通过平台组件和设计 token 维护；业务页面只传标题、数值、趋势、图标名和类型，不在页面 scoped CSS 中单独写统计卡图标尺寸。

模块标题、图表标题、预警标题、列表标题等可复用标题区默认优先使用 `PlatformSectionTitle` 或已接入该组件的 `PlatformSection` / `PlatformEchartsPanel`，不在业务页面重复手写标题字号、字重、右侧补充信息和响应式排列。

二级页面顶部的“页面标题 + 描述 + 右侧动作区/视图切换”默认优先使用 `PlatformViewToolbar`。标准工具按钮走 `tools`，可增减的业务按钮优先走配置化 `actions`，上传、下拉菜单、确认框等特殊交互再使用插槽兜底，不在业务页面重复手写整块 header。

卡片类组件鼠标移入交互默认只做上移反馈，不额外出现品牌描边；如确需 hover 描边，必须说明业务原因并优先沉淀到平台组件或 token。

文件列表、附件列表、文档台账、下载条目等通用文件行场景默认优先使用 `PlatformFileList` / `PlatformFileItem`；业务页面只传文件数据和操作事件，不在页面 scoped CSS 中重复维护文件图标、条目边框、hover 背景和下载按钮布局。

所有数据录入控件必须显式提供符合业务语义的暗文本：输入框使用“请输入…”；下拉框、日期、项目、岗位、状态等选择类控件使用“请选择…”。不得只留下空白输入框或通用“请选择”占位。

弹窗、抽屉或页面内容中出现“基础信息 / 招采信息 / 进场信息”等同级分段切换时，默认优先使用 `PlatformSegmented`，不在业务页面手写 tab/segment 样式。

平台弹窗的内容区与 footer 间距默认由 `PlatformModal` 统一控制，业务页面不得再通过局部 `ant-modal-body`、表单栅格或页面 scoped CSS 追加同类内容边距。

平台弹窗 / 抽屉组件改造时，必须优先排查真实生效层级：`PlatformModal` / `PlatformDrawer` 源组件、共享标题头/图标层、ant-design-vue 语义样式层，以及业务页面残留的 header / body / 分隔线 / margin / padding。禁止只改顶层 class 或单一 padding 后就判断“已生效”。

凡是修改 `PlatformModal`、`PlatformDrawer`、共享标题头、弹窗/抽屉间距、标题区、内容区、footer、全屏逻辑等平台覆盖层，交付前必须至少做一次真实页面浏览器自检，确认实际 DOM 渲染结果与预期一致；如果浏览器实测未完成，不得直接下结论说“已经通过”或“明确生效”。

### 4.3 组件映射

开发或修改前，按任务复杂度输出组件映射：

| 页面区域 | 原生组件 | 平台组件/适配层 | 是否改造 | 样式落点 |
| --- | --- | --- | --- | --- |
| 查询区 | `a-form` / `a-input` / `a-select` | `PlatformSearchForm` 等 | 是/否 | 平台组件/token |
| 表格 | `a-table` / Vxe Grid | `PlatformTable` / Vxe 适配层 | 是/否 | 平台组件/适配层 |
| 树结构 | `a-tree` | `PlatformTree` / `PlatformTreePanel` | 是/否 | 平台组件 |
| 弹窗/抽屉 | `a-modal` / `a-drawer` | `PlatformModal` / `PlatformDrawer` | 是/否 | 平台组件 |

简单样式反馈可以输出简版映射；复杂页面、截图复刻、平台级改造必须输出完整映射和文件归属。

新增平台组件或扩展已有平台组件能力前，必须单独说明：为什么不能复用现有组件、影响哪些页面、是否有页面局部实现替代方案、是否需要 token 或全局样式支持，并等待用户确认。用户说“确认开发”只代表可以进入下一步，不代表允许跳过组件映射、规则自检或私自新增平台组件。

即使是“一行两个”“调宽一点”“改 hover”这类小反馈，只要修改点落在 `components/platform`、适配层、token 或全局样式入口，也必须先说明这是平台能力扩展，并给出简版组件映射和影响范围。

### 4.4 验证要求

修改平台组件后，至少说明：

1. 当前典型页面是否生效。
2. 真实业务页是否已接入，哪些还未接入。
3. 是否有页面绕过平台组件直接使用 ant-design-vue 原生组件。
4. 是否存在页面级样式覆盖导致平台样式不生效。
5. 如果未做浏览器验证、构建或类型检查，说明原因。

### 4.5 浏览器验证隔离规则

开发过程中需要打开页面、截图、验证 UI 时，默认不要控制用户正在使用的 Mac Chrome 浏览器，避免打断用户当前工作。

优先级如下：

1. 优先使用 Codex 内置浏览器、浏览器自动化环境、Playwright 浏览器或独立浏览器会话进行验证。
2. 如果需要使用真实浏览器，默认优先使用 Safari 做本地页面验证，避免和用户日常使用的 Chrome 冲突。
3. 如果 Safari 不可用、无法满足验证目标，才允许使用 Google Chrome，并且必须优先打开新标签页或独立会话，不复用用户当前工作标签。
4. 不要复用用户当前正在工作的 Chrome 窗口或标签页。
5. 不要自动切换、刷新、覆盖用户当前 Chrome 页面。
6. 只有在内置浏览器、Safari 和独立浏览器都不可用时，才允许请求使用系统 Chrome 的现有工作窗口。
7. 使用系统 Chrome 的现有工作窗口前必须先说明原因，并等用户确认。

执行要求：

1. 浏览器验证仍然要做，但必须尽量和用户日常浏览器隔离。
2. 截图验证、路由验证、UI 回归都优先走隔离浏览器；真实浏览器优先 Safari，其次 Chrome 新标签页。
3. 如果无法做到隔离，先停下来说明，不要直接操作用户的 Chrome。

### 4.6 本地预览快速打开规则

本项目本地前端预览默认不是 `http://localhost:5000/`。`5000` 端口在 macOS 上可能被 AirTunes 占用并返回 `403 Forbidden`，不要把它当作 Vite 预览地址反复排查。

每次需要打开或验证页面时，按以下顺序快速处理：

1. 先检查 `http://127.0.0.1:5173/`，必要时再检查 `http://127.0.0.1:5174/`。
2. 如果 `5173/5174` 都不可达，优先在 `apps/web-antd` 目录启动本地 Vite：
   ```bash
   ../../node_modules/.bin/vite --mode development
   ```
3. 如果 `pnpm` 不可用，不要卡在安装或包管理器排查；本仓库已有 `node_modules` 时，直接使用上面的本地 Vite 二进制。
4. Vite 启动后，以终端实际输出的 `Local` 地址为准，通常是 `http://localhost:5173/`；浏览器访问优先写成 `http://127.0.0.1:5173/`。
5. 常用验证页优先打开：
   - `http://127.0.0.1:5173/workbench/index`
   - `http://127.0.0.1:5173/platform/typical-page`
6. 如果 Codex 内置浏览器自动化通道不可用，但用户只是要看页面，先给出可访问地址和服务状态，不要围绕 IAB backend 长时间诊断。
7. 本项目当前固定预览端口就是 `5173`。用户在本仓库内说“打开浏览器”“打开页面”“打开本地预览”等未额外指定端口或地址的口令时，默认直接按 `http://127.0.0.1:5173/` 处理：先检查服务，未启动就立即拉起 `apps/web-antd` 的 Vite，再直接执行打开，不重复向用户确认同一动作，也不要要求用户反复重复同一句口令。

## 5. 页面、截图与 Figma 需求

当用户提供截图、Figma 节点、页面描述，或要求分析列表页、查询页、表单页、弹窗、抽屉、详情页时，先做结构化分析，再开发。

### 5.1 页面分析维度

优先拆解：

1. 顶部区域：标题、按钮、导航、全局操作。
2. 查询筛选区：表单字段、查询、重置、联动。
3. 列表展示区：表格、树表、分页、排序、选择列。
4. 操作区：编辑、删除、查看、导入、导出、批量操作。
5. 弹窗/抽屉：新增、编辑、查看、提交校验、取消关闭。
6. 状态：加载、空态、禁用、选中、错误、权限不可见。

### 5.2 输出强度

1. 简单组件反馈：输出问题归因、组件映射、建议落点，确认后修改。
2. 复杂页面或截图开发：输出页面结构表、组件映射表、文件归属表、交互状态、风险和验证方案。
3. 所有表格必须采用 Markdown 格式。
4. 分析内容必须清晰、结构化，可直接沉淀到项目文档。
5. 截图分析完成后，必须等待用户确认，再进入页面开发或代码修改。
6. 信息不足以判断业务含义、数据来源、权限规则或交互边界时，先提问确认。

### 5.3 截图开发前规则自检

用户提供截图、原型或 Figma 后，即使用户已经确认“开发”，也必须先完成开发前规则自检，再进入代码修改。

必须输出并等待用户确认：

1. 判断本次需求只是修正局部布局，还是需要完整还原截图内容。
2. 判断是否优先复用已有平台组件、适配层、token 或全局样式入口。
3. 判断是否需要新增平台组件；未经用户确认，不得私自新增平台组件。
4. 判断哪些样式属于页面布局，哪些样式应下沉到平台组件或 token。
5. 判断是否会在页面内大量手写 CSS；如超过页面布局所需，必须说明原因、范围和后续迁移计划。
6. 判断页面是否属于查询列表页模式；如果不是，必须说明为什么不使用查询区 + 表格 + 分页结构。
7. 输出组件映射表、文件归属表和规则自检表。

截图内容不得默认完整复刻为业务数据。用户只指出某个问题时，应优先修正该问题，并明确是否需要同步还原其他内容。

台账、记录、管理列表等页面默认先评估“查询区 + 工具栏 + 表格/列表 + 分页 + 批量操作”的结构；如果不采用该模式，必须写明原因并等待用户确认。

所有页面分析优先使用 Vben Admin + ant-design-vue 体系，不绕过平台组件体系。

## 6. 典型页面 Demo 规则

`/platform/typical-page` 的目标是把真实业务页面中的业务内容抽取成平台组件验证场，用于暴露 ant-design-vue 组件平台化问题。它不是虚构页面，也不是简单复刻某个页面源码。

1. 数据、字段、菜单、树节点、表格列、操作入口必须来自当前真实业务页面、路由、接口模型或已存在 Mock。
2. 后端不可用时，可以使用当前项目已有模型、字段、路由、页面配置和已存在 Mock 作为受控数据来源，但必须说明来源。
3. 典型页专用数据源应放在页面专用适配文件中，避免污染真实业务页和全局 mock server。
4. 典型页主体只依赖列表、树、保存、删除、状态切换等抽象方法，不依赖 Mock 实现细节，方便后续切回真实接口。
5. 组件实现必须基于 ant-design-vue 原生组件及平台薄封装；真实业务页当前大量使用 `useVbenVxeGrid`，不能未经确认直接用 `PlatformTable` 批量替换。
6. 顶部导航、左侧导航、面包屑优先在 Vben 布局、菜单源逻辑、布局样式或对应导航组件中处理，不在典型页手写。

当前预览阶段只保留“平台组件”模块作为唯一可见一级模块，默认首页指向 `/platform/typical-page`；“概览”“关于”“工作台”“更新记录”等 Vben 示例模块不作为当前交付范围。

## 7. Vben 布局保护规则

Vben Layout / Menu / Breadcrumb / Tabs / Route-to-Menu 属于后台框架地基层，不为单个页面随意修改。

高风险目录：

1. `packages/effects/layouts/**`
2. `packages/@core/ui-kit/menu-ui/**`

涉及顶部导航、左侧菜单、breadcrumb、tabs、keep-alive、mixed menu、route-to-menu、权限菜单时，先按以下顺序排查：

1. 页面使用方式
2. 路由配置 / menu meta
3. mock 菜单源
4. layout / menu / theme 配置
5. token / CSS 变量
6. 平台组件或项目侧适配层
7. Vben 核心源码

如果必须修改 Vben 核心布局或菜单逻辑，先输出问题归因、影响范围、回滚方案和验证方案，确认后再开发。

## 8. Figma MCP Go 连接规则

本项目使用用户本机的非官方 `@vkhanhqui/figma-mcp-go`，不是官方 Figma MCP。Figma 插件固定监听 `127.0.0.1:1994`，插件面板显示 `Connected` 后，Codex 才能读取当前文件、页面和选中节点。

已验证成功的 Codex MCP 配置是直接执行本机缓存的 darwin arm64 二进制：

```bash
codex mcp remove figma-mcp-go
codex mcp add figma-mcp-go -- /Users/zhuguangxiang/.npm/_npx/ae527a30157df656/node_modules/@vkhanhqui/figma-mcp-go/bin/darwin-arm64/figma-mcp-go
codex mcp get figma-mcp-go
```

排查顺序：

1. 确认 Figma 插件面板显示 `127.0.0.1:1994 Connected`。
2. 执行 `codex mcp get figma-mcp-go`，确认 command 是否指向本机二进制。
3. 如路径失效，先在 `~/.npm/_npx` 下查找 `@vkhanhqui/figma-mcp-go/bin/darwin-arm64/figma-mcp-go`，再重新 `codex mcp remove/add`。
4. 如出现 `localhost` 解析到 `::1` 或 `port taken but leader not responding`，继续优先使用直连二进制方案，并确认插件端固定 `127.0.0.1:1994`。
5. 完成配置后，用 Figma 工具读取 `get_metadata`、`get_selection` 验证连接。

## 9. 前端静态开发与 Mock 模式规则

当前项目阶段主要用于维护平台组件库、产出前端静态交互页面源码，并在后续交付给前端开发与后端进行接口联调。因此，当前阶段不得因为真实后端接口未联通而阻塞进入系统或页面开发。

### 9.1 登录与权限默认要求

1. 必须保留登录页，不允许删除登录页来绕过问题。
2. 必须优先复用 Vben Admin 既有登录、token、用户信息、权限、菜单、动态路由和路由守卫机制。
3. 当 `VITE_USE_MOCK=true` 或项目等价 Mock 配置开启时，登录流程必须使用前端假数据完成。
4. Mock 模式下应提供假 token、假用户信息、假角色权限、假菜单和假路由数据。
5. 页面刷新后，如果本地已有 Mock token，应保持登录状态，不应因为真实后端接口失败退回登录页。

### 9.2 接口与页面开发边界

1. 当前阶段不对接真实后端登录接口、用户信息接口、权限接口或菜单接口。
2. 不要求用户提供后端接口地址。
3. 不因为接口 404、500、跨域、token 失效或菜单接口失败阻塞静态页面开发。
4. 后续页面数据统一优先使用 Mock 数据和前端本地状态模拟。
5. 表格、新增、编辑、删除、详情、筛选等交互，当前阶段优先做前端模拟。
6. Mock 数据刷新后不要求持久化，除登录状态恢复所需 token 和基础状态外，不额外引入持久化复杂度。

### 9.3 切换真实接口规则

1. 切换真实后端时，优先通过环境变量关闭 Mock 模式，例如 `VITE_USE_MOCK=false`。
2. 不为 Mock 模式大规模重构框架底层。
3. 不引入新的 UI 组件库或独立权限体系来解决登录问题。
4. Mock 数据结构应尽量贴近真实接口返回，降低后续联调迁移成本。
5. 如果 Mock 逻辑只是临时兜底，必须在交付报告中说明位置、影响范围和切换方式。

### 9.4 交付预览构建规则

1. 前端联调交付包的 `preview/` 必须使用专门的 handoff 环境构建，不要直接拿 `production` 产物改路径。
2. 构建时必须提供 `apps/web-antd/.env.handoff`，并通过 `npm_lifecycle_script='vite build --mode handoff'` 触发封装插件读取 handoff 配置。
3. 若不按 handoff 模式构建，`preview/` 容易回落到生产接口前缀和绝对资源路径，导致双击预览卡在启动页。
4. 给项目经理和客户演示的交付包，主入口必须优先考虑 Windows 电脑：提供无需开发环境、无需手动打开终端的双击启动器。
5. 不再把 `file://` 直接打开 Vue/Vite SPA 当作主预览方案；`index.html` 只做备用说明入口，真实预览必须通过本地 HTTP 服务进入登录页。
