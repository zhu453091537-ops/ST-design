# 决策记录

本文件记录项目长期有效的关键决策、原因和影响。临时待办写入 `docs/todo-next.md`，阶段性完成记录写入 `docs/project-log.md`。

## 2026-05-05

### 决策 1：Codex 默认作为平台建设战略顾问协作

**决策内容**

在本项目中，Codex 的默认角色不是单纯的代码助手，而是平台建设战略顾问。收到需求后，应优先理解业务目标、分析可行性、整理架构思路、提出技术选型和开发策略；当用户意图不明确时，必须先提问确认。

**原因**

本项目的目标偏向平台建设和长期演进，单点页面实现容易造成规则分散、组件割裂和后续维护成本上升。先分析再实现，可以让每次开发更贴近平台能力建设。

**影响**

1. 不确定需求时，不直接生成代码。
2. 方案明确并得到用户确认后，再进入具体开发建议、代码示例或实际修改。
3. 后续任务需要兼顾业务目标、平台复用性、技术约束和演进成本。

### 决策 2：重要上下文必须沉淀到仓库文档

**决策内容**

项目重要上下文不能只存在聊天记录里。每次开始新任务、接续旧任务、切换电脑、重开聊天，或用户说“开工继续”时，必须先读取：

1. `AGENTS.md`
2. `docs/project-log.md`
3. `docs/decision-records.md`
4. `docs/todo-next.md`

读取后，先输出项目接续摘要，再按用户确认推进。

**原因**

用户会在多台电脑之间切换开发，也可能因为聊天上下文过长而重新开启新聊天。把规则、日志、决策和待办分散在聊天记录里，会导致后续任务无法稳定接续。

**影响**

1. 未读取关键文档前，不允许直接开始大规模修改代码。
2. 阶段完成后必须更新项目日志和下一步待办。
3. 长期规则、技术决策、组件规范、交付规则、禁止事项或平台建设方向，必须同步更新 `docs/decision-records.md` 和 `AGENTS.md`。

### 决策 3：项目文档按职责分工维护

**决策内容**

项目接续文档按职责分工维护：

1. `AGENTS.md`：项目长期规则、开发约束、Codex 执行规范。
2. `docs/project-log.md`：阶段性工作日志，记录已经做过什么。
3. `docs/decision-records.md`：长期有效的关键决策和原因。
4. `docs/todo-next.md`：下一步任务、未完成事项、风险提醒。

**原因**

分开记录长期规则、历史日志、关键决策和当前待办，可以避免文档混杂，方便跨设备和新聊天快速恢复上下文。

**影响**

1. 禁止把临时问题、长期规则、当前待办混在同一个文档里。
2. 后续阶段结束时，需要按内容性质写入对应文档。
3. 用户输入“今天结束”时，必须更新项目日志和下一步待办，并在产生长期规则时同步更新决策记录或 `AGENTS.md`。

### 决策 4：ant-design-vue 组件样式修改默认按平台级改造处理

**决策内容**

本项目基于 Vben Admin 5.0 + ant-design-vue。当用户提出修改 Button、Table、Tree、Form、Input、Select、Modal、Drawer、Tabs、Pagination、SearchForm、列表页、筛选区、弹窗、抽屉等组件样式时，默认理解为基于 ant-design-vue 原生组件进行平台级封装与样式改造，而不是只修改当前业务页面。

**原因**

这些组件通常属于后台管理系统的高频基础能力。如果只在单个页面写局部 CSS，后续会造成样式分散、页面体验不一致、平台组件无法复用、维护成本上升。平台化改造可以统一组件视觉、交互、token、变量和展示页验证。

**影响**

1. 不允许修改 ant-design-vue 源码或 `node_modules`。
2. 不允许引入 Arco、Element Plus、Naive UI 等其他 UI 组件库来替代当前基础组件体系。
3. 不允许只在当前页面写局部 CSS 解决通用组件问题。
4. 平台组件内部可以继续使用 ant-design-vue 原生组件，但业务页面必须优先引用平台组件。
5. 开发或修改前，需要先输出组件映射表，说明页面模块、原生组件、平台组件、是否已有、是否需要改造和样式落点。
6. 如确实只属于当前页面特殊样式，必须先询问用户确认；用户未明确确认“只改当前页面”前，默认按平台级组件改造处理。
7. 修改平台组件后，必须验证典型页面、平台组件展示页、已引用业务页面、直接绕过平台组件的页面，以及页面级样式覆盖风险。
8. 组件改造任务交付报告必须说明修改了哪些平台组件、哪些样式是平台级修改、哪些页面受影响、哪些页面还未接入平台组件、是否存在只改当前页面的特殊情况、当前未完成问题和下一步建议。

### 决策 5：截图与页面需求必须先做结构化分析

**决策内容**

当用户提供截图、截图案例、页面描述，或要求分析列表页、查询页、表单页、弹窗、抽屉、详情页等后台管理页面时，必须先输出页面结构分析、组件映射表、文件归属表、规则自检表、交互与状态说明。分析完成后等待用户确认，再进入页面开发或代码修改。

**原因**

后台管理页面通常由页面标题、查询筛选区、表格列表、操作列、弹窗或抽屉组成。如果没有先把结构、组件来源、文件归属和交互状态说清楚，容易出现页面级临时实现、组件重复建设、样式落点错误或与平台组件治理规则冲突。

**影响**

1. 所有截图和页面需求分析必须使用 Markdown 表格输出组件映射和自检结果。
2. 首选 Vben Admin + ant-design-vue 组件体系，不引入其他 UI 组件库。
3. 文件归属必须基于当前仓库实际结构，如 `apps/web-antd/src/views`、`apps/web-antd/src/components`、`packages/styles` 等，不能直接照搬外部示例路径。
4. 如果截图信息不足以判断业务含义、数据来源、权限规则或交互边界，必须先提问确认。
5. 未经用户确认前，不进入页面开发或代码修改。

### 决策 6：平台组件源头统一放在 components/platform（已被决策 42 替代）

**决策内容**

该决策是平台组件早期建设阶段的目录约定。自 2026-05-14 起，平台组件长期源头已调整为 `packages/platform-ui`，并以 `@st/platform-ui` 作为统一消费入口；`apps/web-antd/src/components/platform/index.ts` 仅保留为迁移过渡兼容出口。具体以“决策 42：平台组件改为 workspace 包化复用”和 `docs/platform-package-governance.md` 为准。

**原因**

典型页面驱动平台组件改造需要一个稳定的源头目录和验证入口。如果 Button、Table、Tree、Form、Modal、Drawer、输入控件等薄封装散落在业务页面里，后续无法判断样式问题是否已经回到平台组件源头，也无法让新页面稳定复用。

**影响**

1. 后续通用组件样式问题优先回到 `packages/platform-ui` 下处理。
2. `/platform/typical-page` 只作为组合验证场，不作为最终样式落点。
3. 新增业务页面应优先从 `@st/platform-ui` 引用平台组件。
4. 当前 `PlatformTable` 基于 ant-design-vue `Table` 薄封装；真实业务页大量使用 `useVbenVxeGrid`，表格核心迁移前必须先确认长期边界。
5. 修改平台组件后，需要同步验证 `/platform/typical-page`，并在真实业务页接入后验证对应业务页。

### 决策 7：顶部菜单层级与表格工具栏必须平台源头化

**决策内容**

后台主布局按照典型页面层级组织菜单：顶部导航栏显示一级功能菜单，左侧导航栏显示当前一级功能菜单下的二级、三级菜单。表格常规功能入口设置、刷新、搜索、全屏统一由 `PlatformTableToolbar` 承载。

**原因**

顶部/左侧导航是后台系统框架能力，不属于单个页面。表格右侧常规功能入口也是所有列表页的共性能力，如果在页面内重复手写，会导致按钮默认态、hover、权限扩展和后续交互不一致。

**影响**

1. `header-sidebar-nav` 布局需要拆分菜单数据，顶部只保留一级菜单，左侧显示子菜单。
2. 典型页面和后续列表页需要表格功能区时，应优先使用 `PlatformTableToolbar`。
3. 表格四个常规功能按钮的默认态、hover、focus 等样式从平台组件和全局变量继承。
4. 页面可以保留业务功能按钮数量和权限逻辑，但表格常规工具入口不得在页面内重复手写。
5. 点击有子级的顶部一级菜单时，应默认跳转到该一级菜单下第一个可导航子菜单，使左侧导航同步选中第一个有效菜单项。
6. 表格工具栏业务按钮按“主按钮最左，次要按钮随后”排序；页面传入 actions slot 时也应遵守该顺序，源组件通过样式兜底主按钮优先。
7. 表格全文搜索属于右侧常规工具入口，默认以搜索图标呈现，点击展开输入框，再次点击收起；不要把搜索输入框长期占在业务按钮和工具按钮中间。
8. `PlatformTable` 对 `column.filters` 提供默认平台筛选面板：必须包含“全部”选项，空筛选状态默认选中“全部”；筛选项点击即应用，不再显示 Antdv 默认“重置 / 确定”按钮。
9. 表头筛选面板的选中态和 hover 态背景统一使用 `--st-color-fill-selected`，即品牌绿 10% 透明度；不得只在某个业务页局部覆盖。

### 决策 8：每个步骤完成后必须主动同步方案进度

**决策内容**

每完成一个步骤、一个阶段或一次代码修改后，必须主动输出简短进度报告。报告需要说明当前方案整体进度、已经完成的内容、剩余内容、风险或问题、下一步建议。平台级组件改造还必须说明样式或交互是否已沉淀到源组件、主题变量、全局样式入口或适配层，以及哪些真实业务页还未接入。

**原因**

当前任务属于平台级通用能力建设，涉及 Figma 样式、Ant Design Vue 原生组件、Vben Vxe 表格、导航布局、真实业务页接入等多个阶段。如果只在最终完成时汇报，用户需要反复追问当前进度和剩余范围，不利于跨设备、跨聊天接续。

**影响**

1. 后续每个阶段完成后都要主动汇报，不等用户追问。
2. 汇报不能只写“已完成”或“通过”，必须说明方案进度和剩余内容。
3. 如果没有浏览器验证或只完成文档、审计、局部验证，需要明确说明原因。
4. 该规则适用于所有后续平台级组件、主题、样式、导航、表格、表单、树结构和页面接入任务。

### 决策 9：真实浏览器验证默认优先 Safari，Chrome 只作隔离回退

**决策内容**

当任务需要在真实浏览器中打开本地页面、截图或验证 UI 时，默认优先使用 Safari；只有 Safari 不可用或无法满足验证目标时，才允许使用 Google Chrome，并且应优先打开新标签页或独立会话，不复用用户正在工作的 Chrome 标签页。

**原因**

用户长期在 Chrome 中同时处理开发、设计和素材管理工作，直接复用现有 Chrome 标签页容易打断当前工作流，尤其在跨多个系统标签页切换时冲突明显。将 Safari 作为真实浏览器验证的默认首选，可以显著降低对用户工作现场的干扰。

**影响**

1. 后续真实浏览器验证优先级调整为：内置/自动化浏览器 → Safari → Chrome 新标签页/独立会话 → 经确认后再动用户当前 Chrome 工作窗口。
2. 除非用户明确同意，不再优先操作用户当前 Chrome 标签页。
3. 若必须使用 Chrome，也要先采用新标签页或独立会话，不直接刷新、覆盖或切走用户当前工作内容。

### 决策 9A：同一视觉问题连续两次未解时，第三次前必须主动浏览器自检

**决策内容**

对于同一个 UI / 样式 / 交互问题，如果已经连续修改 2 次仍未得到用户明确确认“已解决”，则第 3 次开始前必须主动执行真实浏览器自检，再决定下一步修改点。不得继续只凭代码 diff、静态阅读或口头推断反复试错。

**原因**

平台组件问题经常存在“代码改到了，但真实生效层没命中”或“改动被其他层覆盖”的情况。若连续两次未解还不及时自检，就容易进入错误层级的反复试错，既拖慢进度，也损害用户对结果判断的信任。

**影响**

1. 同类问题从第 3 次尝试起，浏览器自检不再是“可选验证”，而是继续修改前的硬前置动作。
2. 后续汇报必须明确区分“已做浏览器自检”和“尚未做浏览器自检”，未自检前不得直接下结论说“已经好了”“已经生效”或“应该没问题”。
3. 该规则尤其适用于 `Table`、`Tree`、`Form`、`Modal`、`Drawer`、`Select`、`Menu` 等平台通用组件的视觉与交互问题。

### 决策 9：树筛选容器按平台壳下沉，业务页保留数据与联动逻辑

**决策内容**

对于后台管理页高频出现的“左侧树筛选 + 搜索 + 刷新 + 空态 + 骨架屏”结构，优先沉淀到 `apps/web-antd/src/components/platform/tree/platform-tree-panel.vue` 作为平台壳。真实业务页继续保留树数据请求、筛选联动、查询参数组装、`reload/select` 事件和业务插槽内容。

**原因**

`system/user/dept-tree.vue` 和典型页都存在相同的树筛选结构。如果只在单个页面里组合 `Tree`、`Input`、刷新按钮和空态，会让树结构样式、搜索区间距、按钮状态和后续交互扩展继续分散。把“壳”下沉到平台层，可以统一视觉和交互，同时不强行改动业务页的接口、权限和联动逻辑。

**影响**

1. 树筛选的头部结构、搜索框、刷新按钮、骨架屏和空态优先回到 `PlatformTreePanel` 维护。
2. `PlatformTree` 继续作为树本体样式薄封装，容器级组合能力由 `PlatformTreePanel` 承担。
3. 真实业务页接入树筛选平台化时，应优先复用 `PlatformTreePanel`，而不是在页面内重复手写 `Tree + Input + Button`。
4. 树数据加载、过滤、节点点击联动和查询参数拼装仍保留在业务页面，避免平台组件侵入业务逻辑。

### 决策 10：当前预览只保留平台组件模块

**决策内容**

当前前端预览阶段只保留“平台组件”模块作为唯一可见一级模块，默认首页统一指向 `/platform/typical-page`。Vben 示例的“概览”“关于”“工作台”“更新记录”等模块不再作为当前预览交付范围。

**原因**

当前工作重心是平台组件源头、典型页面验证场和真实业务页接入治理。保留 Vben 示例模块会干扰用户判断当前交付范围，也会让顶部导航和左侧导航出现不需要的入口。

**影响**

1. Mock 菜单源只保留 `Platform` 和 `PlatformTypicalPage`。
2. 前端路由增加平台组件模块，确保混合权限模式下平台组件路由可稳定生成。
3. 旧 `/analytics`、`/workspace`、`/changelog`、`/vben-admin/about` 等地址只保留隐藏重定向到 `/platform/typical-page`，不再显示为可见菜单。
4. 后续如需恢复业务模块，应按真实业务范围重新接入，而不是恢复 Vben 示例模块作为默认导航。

### 决策 11：Figma 连接默认使用本地非官方 Figma MCP Go

**决策内容**

本项目 Figma 连接默认使用用户本机的非官方 `@vkhanhqui/figma-mcp-go`，插件固定连接 `127.0.0.1:1994`，不默认使用官方 Figma MCP。本轮已验证成功的 Codex MCP 配置为直接执行本机缓存的 darwin arm64 二进制：

```bash
codex mcp remove figma-mcp-go
codex mcp add figma-mcp-go -- /Users/zhuguangxiang/.npm/_npx/ae527a30157df656/node_modules/@vkhanhqui/figma-mcp-go/bin/darwin-arm64/figma-mcp-go
codex mcp get figma-mcp-go
```

**原因**

用户明确说明当前使用的不是官方 MCP，且 Figma 插件固定为 `127.0.0.1:1994`。之前通过 `npx -y @vkhanhqui/figma-mcp-go` 间接启动时，出现 `localhost`、`::1` 和服务角色判断相关问题，导致插件显示 `Disconnected`。改为让 Codex MCP 直接执行本机二进制后，插件已显示 `Connected`，并成功读取 Figma 文件、页面和选中节点。

**影响**

1. 后续 Figma 任务先检查 `figma-mcp-go` 本地连接，不要默认切换官方 MCP。
2. Figma 插件必须显示 `127.0.0.1:1994 Connected` 后，再读取节点、截图或设计上下文。
3. 如二进制缓存路径失效，应先在 `~/.npm/_npx` 下重新查找 `@vkhanhqui/figma-mcp-go/bin/darwin-arm64/figma-mcp-go`，再更新 Codex MCP 配置。
4. 若再次出现 `Disconnected`，按 `AGENTS.md` 第 8 节排查。

### 决策 12：平台弹窗内容区与 footer 间距统一收口到 `PlatformModal`

**决策内容**

`PlatformModal` 统一负责平台弹窗的标题、内容区和 footer 间距，包含内容区左右 40px、footer 分隔线和按钮区内边距。业务页面不得再单独通过 `.ant-modal-body`、表单网格或页面 scoped CSS 叠加同类内容边距。

**原因**

弹窗内容区与 footer 属于平台级统一视觉结构，不应随单个页面分散维护。若业务页各自补 padding，会导致同一平台弹窗在不同页面出现不同的内容节奏，破坏一致性，也会让后续修改无法“一处修改，全局生效”。

**影响**

1. 后续平台弹窗内容边距优先改 `PlatformModal`，不再写页面级补丁。
2. 业务页表单栅格只负责布局和列间距，不再承担平台弹窗的外边距职责。
3. 新页面如果接入 `PlatformModal`，默认继承统一的标题、内容和 footer 节奏。

### 决策 14：客户演示交付包采用 Windows 优先双击启动器

3. 前端联调仍以 `source/` 完整源码为准，不能只交 `preview/` 或只交 `dist`。
4. README 必须把“客户演示入口”和“前端联调源码入口”分开说明。

### 决策 16：Vben 核心布局需先做分层归因后再改动

**决策内容**

**影响**

2. 不允许未经确认就大改 `use-mixed-menu.ts`、route → menu 逻辑、breadcrumb、tabs / keep-alive。

4. 通过全局覆盖 `.ant-menu`、`.ant-layout` 粗暴修改导航样式要被视为高风险行为。
5. 如果确实需要修改 Vben 核心布局或菜单逻辑，开发前必须先输出问题归因、影响范围、回滚方案和验证方案。

### 决策 19：当前阶段默认使用 Mock 登录与前端静态交互

### 决策 21：本地预览默认走 5173，不再把 5000 当作项目端口

**决策内容**

本项目本地预览默认端口以 `5173` 为准，必要时回退到 `5174`。`http://localhost:5000/` 不再作为本项目的默认预览入口，也不要把它当作 Vite 启动失败的优先排查对象。

**原因**

实际排查中，`5000` 在当前 macOS 环境会被系统服务占用并返回 `403 Forbidden`，会浪费大量时间；而本项目的 Vite 开发服务实际由 `apps/web-antd` 启动后监听 `5173/5174`。把默认端口固定到正确位置，可以减少每次打开浏览器时的无效绕路。

**影响**

1. 打开本地页面时，优先使用 `http://127.0.0.1:5173/`，必要时检查 `http://127.0.0.1:5174/`。
2. 若 `5173/5174` 不可达，优先在 `apps/web-antd` 目录用本地 Vite 二进制启动，而不是先查 `5000`。
3. 如果 `pnpm` 不可用，不应因此阻塞页面打开；仓库已有 `node_modules` 时可以直接使用 `../../node_modules/.bin/vite --mode development`。
4. Codex 内置浏览器不可用时，先给出实际可访问地址和服务状态，不围绕 IAB backend 做长时间诊断。

### 决策 22：统计卡图标尺寸必须平台化并由 token 驱动

**决策内容**

`PlatformStatCard` 的右上角图标尺寸必须由平台组件统一控制，默认外层图标容器为 `48px` 方形，内层图标尺寸为 `2rem`；业务页面不得把这类尺寸散落到页面 scoped CSS 里。若后续确有特殊场景，可以通过组件 props 覆盖，但默认值仍应来自设计 token。

**原因**

统计卡是项目总览页中高频复用的总览信息组件。图标容器和图标本体尺寸如果分散写在页面里，会导致后续不同页面难以统一调整，也不利于继续做平台化沉淀。

**影响**

1. `packages/@core/base/design/src/design-tokens/default.css` 中新增统计卡图标尺寸 token。
2. `apps/web-antd/src/components/platform/stat/platform-stat-card.vue` 统一读取 token，业务页面只传内容数据。
3. 后续同类统计卡样式调整优先回到 `PlatformStatCard`，不在页面内重复写图标尺寸。

### 决策 23：平台表格默认按页码完整展示并由页面整体滚动

**决策内容**

`PlatformTable` 默认不根据浏览器高度计算表格 body 的内部滚动高度。页码器当前页要求展示多少条数据，表格就在当前页面完整渲染多少条；如果内容超出浏览器可视高度，由页面整体上下滚动，而不是优先让表格 body 内部滚动。

**原因**

用户明确希望后台查询列表页遵循更接近成熟平台表格的交互：一页展示多少条，就完整看到多少条；当内容很多时，由页面整体滚动，而不是把滚动条锁进表格内部。这样分页感知更直接，也更符合截图参考的使用习惯。

**影响**

1. `PlatformTable` 默认关闭基于浏览器高度的内部滚动自适应。
2. 调用方如果显式传入 `scroll.y`，或在开发时被明确要求启用表格内部滚动，组件才进入内部滚动模式。
3. 普通列表页不在业务页面 scoped CSS 中硬编码表格高度，也不再默认依赖表格 body 内部滚动。

### 决策 24：平台表格外描边颜色由专用 token 控制

**决策内容**

`PlatformTable` 的表格外描边由 `--st-color-table-outline` 控制，并在源组件中作用于 `.ant-table-container`。该规则只属于平台表格组件，不在 `/workbench/index`、`/project/information` 等业务页面写局部边框样式，也不全局覆盖所有原生 Antdv 表格。

**原因**

截图示例体现的是表格整体容器需要清晰的外描边。外描边属于表格原组件视觉规范，如果写在当前页面，会导致后续同类平台表格无法统一继承。

**影响**

1. `packages/@core/base/design/src/design-tokens/default.css` 新增 `--st-color-table-outline`，默认读取 `--st-color-border-control`。
2. `packages/@core/base/design/src/design-tokens/dark.css` 覆盖 `--st-color-table-outline` 为暗色主题 `--border`。
3. `apps/web-antd/src/components/platform/table/platform-table.vue` 在组件作用域内给 `.ant-table-container` 加 1px 外描边。

### 决策 25：平台表格左右边缘单元格内边距统一为 24px

**决策内容**

`PlatformTable` 的第一列和操作列单元格默认使用 `--st-table-edge-cell-padding: 24px` 作为左右内边距。该规则同时作用于表头和表体边缘单元格，确保表头文字与表体内容保持同列对齐。平台表格的操作列默认宽度为 `184px`，避免操作列被业务页配置撑得过宽；确有特殊操作数量时，可通过 `actionColumnWidth` 覆盖。固定列存在时，平台表格会按叶子列宽自动合成 `scroll.x`，避免横向滚动时表头和表体使用不同滚动模型。

**原因**

表格左右边缘是列表页扫描时最敏感的位置。如果只在当前页面调表头，会造成后续平台表格边缘留白不一致；如果只调表头不调表体，会造成列内容视觉错位。

**影响**

1. `packages/@core/base/design/src/design-tokens/default.css` 新增 `--st-table-edge-cell-padding` 和 `--st-table-action-column-width`。
2. `apps/web-antd/src/components/platform/table/platform-table.vue` 在组件作用域内处理首列 `th/td` 和平台操作列。
3. `PlatformTable` 会识别 `key === 'action'` 或标题为“操作”的列，添加专用 class，并使用平台默认操作列宽度。
4. 固定列存在且调用方未显式传入 `scroll.x` 时，`PlatformTable` 自动按列宽合成横向滚动宽度。
5. 仅影响已接入 `PlatformTable` 的表格；Vxe Grid 或直接使用 Antdv Table 的页面不自动继承。

### 决策 26：平台模块默认间距和内容内边距统一为 24px

**决策内容**

平台模块 / 页面区块之间的默认间距统一为 `--st-layout-section-gap: 24px`；模块内容区默认内边距统一为 `--st-module-content-padding: 24px`。这两个变量定义在设计 token 中，并由 `Page` 默认内容容器、`.platform-surface`、`PlatformTableToolbar`、`PlatformStatCard` 和已接入平台模块的项目页面优先使用。

**原因**

项目总览和项目信息管理页面已暴露出默认模块间距偏小的问题。如果继续在页面 scoped CSS 中写 `16px`，后续不同业务页会出现区块密度不一致，也不利于统一调整平台视觉节奏。

**影响**

1. 页面默认内容 padding 不再依赖 Tailwind `p-4`，改为读取 `--st-module-content-padding`。
2. 通用模块容器 `.platform-surface` 默认拥有 24px 内容内边距。
3. `PlatformTableToolbar` 的 bleed 偏移和左右内边距跟随模块内容内边距，上下内边距保持 `12px`，避免工具条高度过厚。
4. 后续新增平台列表页、查询区、表格区、详情区时，应优先读取这两个变量；特殊紧凑布局需要说明原因。

### 决策 27：文件列表和文件条目进入平台组件层

**决策内容**

`项目全景管理 - 文档与台账管理` 中的文件列表不使用业务页面大段 scoped CSS 临时实现，而是新增 `PlatformFileList` 和 `PlatformFileItem` 作为平台文件列表薄封装。业务页面只传文件名、项目名、大小、日期、类型和下载事件，文件图标、条目边框、hover 背景、下载按钮布局等统一由平台组件维护。

**原因**

合同附件、技术方案、验收报告、台账清单等文件条目会在多个业务页面复用。如果每个页面单独写文件行样式，后续会出现图标、间距、按钮和 hover 状态不一致。把文件列表下沉到平台组件，可以让当前文档页和后续附件/台账场景复用同一套视觉与交互。

**影响**

1. 新增平台组件目录 `apps/web-antd/src/components/platform/file`。
2. 平台组件出口 `apps/web-antd/src/components/platform/index.ts` 对外导出文件列表组件和类型。
3. 后续出现“文件列表 / 附件列表 / 文档台账 / 下载条目”场景时，优先复用 `PlatformFileList` 和 `PlatformFileItem`，不在页面内重复写文件条目通用样式。
4. 上传、导出、预览、权限和真实文件接口仍属于业务页或后续接口适配层，不放入文件列表基础展示组件。

### 决策 28：截图开发确认后仍必须先做规则自检

**决策内容**

用户提供截图、原型或 Figma 后，即使已经确认“开发”，Codex 仍必须在代码修改前输出规则自检，并等待用户确认。自检必须覆盖：是否只是局部布局修正、是否需要完整还原截图、是否优先复用平台组件、是否需要新增平台组件、是否会大量手写页面 CSS、页面是否属于查询列表页模式、组件映射表和文件归属表。

**影响**

2. 截图内容不得默认完整复刻为业务数据；用户只指出一个问题时，优先修正该问题，并说明是否同步还原其他内容。

4. 页面内 CSS 只允许处理布局、分栏、间距、高度和滚动区域；超出布局范围的通用视觉必须回到平台组件、token 或全局样式入口。

### 决策 31：数据录入控件必须提供业务化暗文本

**决策内容**

所有新增或调整的表单、弹窗、抽屉、搜索区和页面录入区域中，输入框、下拉框、日期选择、项目选择、岗位选择、状态选择等数据录入控件都必须显式提供 placeholder。输入类控件使用“请输入 + 字段名”，选择类控件使用“请选择 + 字段名”，不得只保留空白输入框、默认空值或通用“请选择”。

3. 后续新增表单字段时，placeholder 属于验收项；遗漏时应在同轮修正。

### 决策 35：查询面板与表格工具栏分层治理

标准查询列表页默认采用 `PlatformQueryPanel + PlatformTableToolbar` 分层方案：`PlatformQueryPanel` 统一承载筛选字段、查询、重置和收起/展开；`PlatformTableToolbar` 继续只负责表格上方业务按钮以及刷新、设置、全屏等通用工具。

**原因**

当前列表页的查询区虽然已经有 `PlatformSearchForm`、平台按钮和平台字段组件可复用，但“字段网格 / 查询动作 / 收起逻辑”仍需要页面自行拼装。如果继续在业务页散写，后续高频列表页会反复复制折叠状态、按钮顺序和筛选区间距，平台治理边界也会继续模糊。

**影响**

1. 后续新增查询列表页时，筛选区优先复用 `PlatformQueryPanel`，不要再在页面层重复组合 `PlatformSearchForm + 按钮组 + 折叠逻辑`。
2. `PlatformQueryPanel` 默认以收起态进入；除非页面在开发时被明确要求默认全部展开，否则统一先展示收起态。
3. `PlatformQueryPanel` 对外表现为一个复合平台组件，但内部仍优先复用现有平台按钮、字段组件和设计 token，避免再引入外部 Pro 体系。
4. `PlatformTableToolbar` 只保留表格业务按钮和通用工具，不再承担页面筛选按钮，防止查询区和表格工具区职责混杂。
5. 首批验证页面为 `/battery/construction`；后续可逐步回收项目、人员、台账和审批列表页的存量查询区实现。

### 决策 36：表格工具栏搜索与任务卡平台边界

**决策内容**

`PlatformTableToolbar` 的右侧标准工具固定按 `search / refresh / setting / fullscreen` 作为默认完整能力；业务按钮继续放在左侧 `actions` 插槽或配置入口中。`PlatformQueryPanel` 不承接关键词搜索，只承接结构化筛选字段、查询、重置和折叠。新增 `PlatformTaskCard` 作为任务 / 待办卡片平台组件，承载标题、描述、状态标签、元信息、进度和主操作。

**原因**

本轮迁移 `PlatformQueryPanel` 时暴露出一个边界误判：如果把“搜索”理解成筛选区能力，表格工具栏就会少掉搜索入口，和平台表格工具区规则不一致。同时 `/project/evaluation` 的待评估项目卡已经具备稳定任务卡结构，继续留在页面内会让同类任务卡的标题字号、标签间距、进度条和操作按钮重复扩散。

**影响**

1. 查询列表页的关键词搜索默认仍放在 `PlatformTableToolbar`，不放进 `PlatformQueryPanel`。
2. 页面显式传入 `tools` 时，不得误删 `search`；如确需隐藏搜索，必须说明页面没有关键词搜索业务。
3. `PlatformTaskCard` 适用于任务、待办、评估发起等“标题 + 标签 + 元信息 + 进度 + 操作”的卡片，不作为所有业务卡片的通用替代。
4. 页面分栏宽度、卡片列表高度和滚动容器仍属于页面布局，不进入 `PlatformTaskCard`。
5. 首个接入页面为 `/project/evaluation`；后续如 `/personnel/worktime` 或人员档案卡出现稳定共性，再单独评估是否扩展平台卡片能力。

### 决策 37：审批进度统一沉淀为平台组件

**决策内容**

施工管理、审批详情等页面中的右侧审批进度展示，应统一沉淀为平台组件 `PlatformApprovalProgress`，不要在业务页面里重新手写时间线、节点状态、头像标识和时间文本布局。

**原因**

审批进度属于高复用的后台通用展示能力，后续在多个页面都会出现。如果每个页面各自拼 `Timeline`，后续统一修改节点颜色、间距、头像样式和状态文本时会非常分散。

**影响**

1. 新增或改造审批进度时，优先修改 `apps/web-antd/src/components/platform/approval/**`。
2. 业务页面只负责传入节点数据和局部布局，不再重复维护右侧审批样式。
3. 后续需要审批进度的页面，应优先复用 `PlatformApprovalProgress`，再按业务需要扩展数据字段。

### 决策 38：登录后默认首页切回项目全景管理的项目总览

**决策内容**

当前平台登录后的默认首页统一切回 `项目全景管理 > 项目总览`，即路由 `/workbench/index`。`/platform/typical-page` 继续保留为平台组件与人员档案管理的验证入口，但不再承担整个系统的默认首页职责。

**原因**

随着真实业务模块逐步接入，`项目总览` 已经承担平台首页角色，用户登录后应优先进入真实业务首页，而不是平台验证页。若继续把默认首页指向典型页，会让首页定位、菜单认知和后续接续文档产生偏差。

**影响**

1. `apps/web-antd/src/preferences.ts` 中的 `defaultHomePath` 默认指向 `/workbench/index`。
2. 根路由 `/`、登录后的默认回跳、以及旧 `dashboard / analytics / workspace / about / changelog` 隐藏入口重定向，都应统一落到 `/workbench/index`。
3. `/platform/typical-page` 仍用于平台组件验证和人员档案管理场景，但后续文档、演示和接续摘要不得再将其描述为“登录后默认首页”。

### 决策 39：登录页默认居中并隐藏认证工具栏

**决策内容**

登录页默认认证布局统一使用 `panel-center`，并在项目层关闭认证工具栏，避免登录页显示颜色切换和布局切换按钮。

**原因**

登录页是进入系统的第一屏，默认居中能保持最稳定的登录视觉；颜色和布局切换属于偏好配置入口，不应出现在默认登录首屏打扰用户。

**影响**

1. `apps/web-antd/src/preferences.ts` 中的 `authPageLayout` 默认值统一为 `panel-center`。
2. `apps/web-antd/src/layouts/auth.vue` 默认关闭 `AuthPageLayout` 的 `toolbar`。
3. 后续如果需要恢复登录页工具栏，只能显式重新开启，不能再依赖默认值自动显示。

### 决策 40：登录成功后默认优先进入项目总览

**决策内容**

登录成功后默认优先进入 `项目全景管理 - 项目总览`，即 `/workbench/index`。登录页 URL 上即使残留了业务页跳转过来的 `redirect` 参数，也不再用它覆盖平台首页的默认落点。

**原因**

当前平台首页已经明确是项目总览。用户退出登录后重新进入系统，应该始终先看到平台首页，而不是被历史业务页或登录前的停留页面带偏到人员模块等次级页面。

**影响**

1. `apps/web-antd/src/router/guard.ts` 中登录成功后的默认回跳优先级改为首页优先。
2. 未登录访问受保护业务页时，仍允许先跳登录页；但登录完成后统一先进入 `/workbench/index`。
3. 如果未来某些页面确实要求“登录后必须回原页”，需要单独做白名单或场景化策略，不能再直接复用当前统一首页逻辑。

### 决策 41：修改平台组件默认先追源码和生效链路

**决策内容**

修改 Table、Modal、Drawer、Menu、Tree、Form、Select 等平台级通用组件时，默认先追源码和生效链路，再决定修改点。推荐顺序是：原组件源码和 `style/token/theme` 定义 -> `ConfigProvider` / theme 注入 -> 平台适配层 -> 全局样式和 token -> 页面级补丁。只有确认“真正生效层”后，才允许动手改代码。

**原因**

这类问题经常不是表面样式没写，而是底层 token、组件内部计算样式或适配层把结果覆盖掉了。先找生效链路能减少来回试错，避免每次都花很久才能找到“底层元凶”。

**影响**

1. 以后遇到“改了没生效”“只在某些页面生效”的问题，必须先查源码和主题注入链。
2. 不再优先用业务页 scoped CSS 试错，再回头找底层。
3. 当问题最终落在平台层时，修改必须同步说明生效层级和影响范围。
4. 这条规则优先适用于 `Table`、`Modal`、`Drawer`、`Menu`、`Tree`、`Form`、`Select` 及其适配层。

### 决策 42：平台组件改为 workspace 包化复用

**决策内容**

平台组件长期从 `apps/web-antd/src/components/platform` 迁移到 `packages/platform-ui`，并以 `@st/platform-ui` 作为统一消费入口。平台样式、适配、类型和工具分别按职责沉淀到 `packages/platform-styles`、`packages/platform-adapter`、`packages/platform-types` 和 `packages/platform-utils`。正式复用方式使用 `pnpm workspace` 与 `workspace:*`，由包管理器自动生成软连接，不手工维护 symlink，不复制多份组件源码。

迁移过程分阶段执行：先建立规范和包骨架，再做无行为变化迁移，保留 `apps/web-antd/src/components/platform/index.ts` 作为兼容出口，最后再逐步切换新增页面和存量页面的导入路径。

**原因**

近期平台组件修改暴露出组件不统一、修改点难追踪和多处页面可能分叉的问题。如果平台组件长期停留在单个业务 app 内，后续多个项目复用时容易出现多份组件拷贝，导致一处修复无法自动影响其它项目。使用 workspace 包可以在不牺牲本地开发体验的前提下保留唯一源码、统一依赖和统一交付边界。

**影响**

1. `packages/platform-ui` 是平台 Vue 组件的目标唯一源码位置。
2. 新页面和新平台能力默认从 `@st/platform-ui` 引用平台组件。
3. `apps/web-antd` 只承担业务页面、业务状态、业务 Mock、路由和交付入口，不再长期维护平台组件第二份源码。
4. 迁移期间不做一次性全量业务页面导入替换，优先保持已开发功能稳定。
5. 给前端和后端联调的源码交付包必须保留 monorepo 结构，不能只交付 `apps/web-antd`。
6. 预览构建产物不受源码包化影响，Vite 会把被引用的平台组件打进最终产物。
7. 具体执行规则以 `docs/platform-package-governance.md` 为准。
8. `platform-styles` 与 `platform-adapter` 可在迁移过渡期先代理现有 Vben 样式和插件入口，先收口引用边界，再分阶段迁移真实实现。
9. `@st/platform-adapter/echarts` 作为 ECharts 适配入口，先无行为变化转发现有 Vben ECharts 插件；平台组件和业务页面不得继续直接新增 `@vben/plugins/echarts` 引用。
10. `@st/platform-adapter/upload` 先承载上传纯类型、默认 accept 列表、默认文件预览函数、`UploadInfoApi` 查询抽象、`UploadFeedbackAdapter` 反馈抽象和裁剪上传契约类型；`FileUpload` / `ImageUpload` 组件、默认 `uploadApi`、默认 `ossInfo`、默认 `window.message/window.modal` 反馈实现、头像裁剪 UI 和业务弹窗仍留在 `apps/web-antd`，未完成业务依赖拆分前不得整包搬入平台适配层。
11. `@st/platform-styles/tokens` 是 ST-design 平台 CSS 变量的长期落点；`packages/@core/base/design/src/design-tokens/*` 只保留 Vben 通用主题变量，并通过 `var(--st-*)` 消费平台 token，不再直接维护平台变量源值。
12. `@st/platform-styles/antd` 只承载 Ant Design Vue 全局覆盖和平台通用 CSS；`@st/platform-styles/vxe-table` 只承载 Vxe 专属覆盖。后续不要把 Vxe 专属样式继续写入 Antdv 入口，也不要把 Antdv 通用覆盖写回 `packages/styles/src/antd/index.css`。
13. `@st/platform-styles/antd` 内部按真实覆盖对象拆分为 `base / button / field / form / tree / table / pagination / overlay / tag / platform-common / feedback / validation / utilities` 等文件，`index.css` 只负责按顺序导入；后续修改 Antdv 覆盖时优先落到对应分文件，不再把新规则堆回入口文件。
14. 平台封装组件遇到 Ant Design Vue 新旧语义属性变更时，优先在 `packages/platform-ui` 做兼容转换，例如 `PlatformModal` / `PlatformDrawer` 将历史 `bodyStyle`、`destroyOnClose`、`width` / `height` 等调用转换为当前 `styles.body`、`destroyOnHidden`、`size` 等语义；业务页面后续不得继续复制废弃属性作为新实现。
