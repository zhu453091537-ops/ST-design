# 决策记录

本文件记录项目长期有效的关键决策、原因和影响。临时待办写入 `docs/todo-next.md`，阶段性完成记录写入 `docs/project-log.md`。

## 2026-05-05

### 决策 1：Codex 默认作为平台建设战略顾问协作

**决策内容**

在本项目中，Codex 的默认角色不是单纯的代码助手，而是平台建设战略顾问。收到需求后，应优先理解业务目标、分析可行性、整理架构思路、提出技术选型和开发策略；当用户意图不明确时，必须先提问确认。

**原因**

本项目的目标偏向平台建设和长期演进，单点页面实现容易造成规则分散、组件割裂和后续维护成本上升。先分析再实现，可以让每次开发更贴近平台能力建设。

**影响**

1. 不确定需求时，不直接生成代码。
2. 方案明确并得到用户确认后，再进入具体开发建议、代码示例或实际修改。
3. 后续任务需要兼顾业务目标、平台复用性、技术约束和演进成本。

### 决策 2：重要上下文必须沉淀到仓库文档

**决策内容**

项目重要上下文不能只存在聊天记录里。每次开始新任务、接续旧任务、切换电脑、重开聊天，或用户说“开工继续”时，必须先读取：

1. `AGENTS.md`
2. `docs/project-log.md`
3. `docs/decision-records.md`
4. `docs/todo-next.md`

读取后，先输出项目接续摘要，再按用户确认推进。

**原因**

用户会在多台电脑之间切换开发，也可能因为聊天上下文过长而重新开启新聊天。把规则、日志、决策和待办分散在聊天记录里，会导致后续任务无法稳定接续。

**影响**

1. 未读取关键文档前，不允许直接开始大规模修改代码。
2. 阶段完成后必须更新项目日志和下一步待办。
3. 长期规则、技术决策、组件规范、交付规则、禁止事项或平台建设方向，必须同步更新 `docs/decision-records.md` 和 `AGENTS.md`。

### 决策 3：项目文档按职责分工维护

**决策内容**

项目接续文档按职责分工维护：

1. `AGENTS.md`：项目长期规则、开发约束、Codex 执行规范。
2. `docs/project-log.md`：阶段性工作日志，记录已经做过什么。
3. `docs/decision-records.md`：长期有效的关键决策和原因。
4. `docs/todo-next.md`：下一步任务、未完成事项、风险提醒。

**原因**

分开记录长期规则、历史日志、关键决策和当前待办，可以避免文档混杂，方便跨设备和新聊天快速恢复上下文。

**影响**

1. 禁止把临时问题、长期规则、当前待办混在同一个文档里。
2. 后续阶段结束时，需要按内容性质写入对应文档。
3. 用户输入“今天结束”时，必须更新项目日志和下一步待办，并在产生长期规则时同步更新决策记录或 `AGENTS.md`。

### 决策 4：ant-design-vue 组件样式修改默认按平台级改造处理

**决策内容**

本项目基于 Vben Admin 5.0 + ant-design-vue。当用户提出修改 Button、Table、Tree、Form、Input、Select、Modal、Drawer、Tabs、Pagination、SearchForm、列表页、筛选区、弹窗、抽屉等组件样式时，默认理解为基于 ant-design-vue 原生组件进行平台级封装与样式改造，而不是只修改当前业务页面。

**原因**

这些组件通常属于后台管理系统的高频基础能力。如果只在单个页面写局部 CSS，后续会造成样式分散、页面体验不一致、平台组件无法复用、维护成本上升。平台化改造可以统一组件视觉、交互、token、变量和展示页验证。

**影响**

1. 不允许修改 ant-design-vue 源码或 `node_modules`。
2. 不允许引入 Arco、Element Plus、Naive UI 等其他 UI 组件库来替代当前基础组件体系。
3. 不允许只在当前页面写局部 CSS 解决通用组件问题。
4. 平台组件内部可以继续使用 ant-design-vue 原生组件，但业务页面必须优先引用平台组件。
5. 开发或修改前，需要先输出组件映射表，说明页面模块、原生组件、平台组件、是否已有、是否需要改造和样式落点。
6. 如确实只属于当前页面特殊样式，必须先询问用户确认；用户未明确确认“只改当前页面”前，默认按平台级组件改造处理。
7. 修改平台组件后，必须验证典型页面、平台组件展示页、已引用业务页面、直接绕过平台组件的页面，以及页面级样式覆盖风险。
8. 组件改造任务交付报告必须说明修改了哪些平台组件、哪些样式是平台级修改、哪些页面受影响、哪些页面还未接入平台组件、是否存在只改当前页面的特殊情况、当前未完成问题和下一步建议。

### 决策 5：截图与页面需求必须先做结构化分析

**决策内容**

当用户提供截图、截图案例、页面描述，或要求分析列表页、查询页、表单页、弹窗、抽屉、详情页等后台管理页面时，必须先输出页面结构分析、组件映射表、文件归属表、规则自检表、交互与状态说明。分析完成后等待用户确认，再进入页面开发或代码修改。

**原因**

后台管理页面通常由页面标题、查询筛选区、表格列表、操作列、弹窗或抽屉组成。如果没有先把结构、组件来源、文件归属和交互状态说清楚，容易出现页面级临时实现、组件重复建设、样式落点错误或与平台组件治理规则冲突。

**影响**

1. 所有截图和页面需求分析必须使用 Markdown 表格输出组件映射和自检结果。
2. 首选 Vben Admin + ant-design-vue 组件体系，不引入其他 UI 组件库。
3. 文件归属必须基于当前仓库实际结构，如 `apps/web-antd/src/views`、`apps/web-antd/src/components`、`packages/styles` 等，不能直接照搬外部示例路径。
4. 如果截图信息不足以判断业务含义、数据来源、权限规则或交互边界，必须先提问确认。
5. 未经用户确认前，不进入页面开发或代码修改。

### 决策 6：平台组件源头统一放在 components/platform

**决策内容**

当前平台组件源头统一放在 `apps/web-antd/src/components/platform`，并通过 `apps/web-antd/src/components/platform/index.ts` 对外提供出口。典型页面验证场放在 `apps/web-antd/src/views/platform/typical-page/index.vue`，Mock 菜单路径为 `/platform/typical-page`。

**原因**

典型页面驱动平台组件改造需要一个稳定的源头目录和验证入口。如果 Button、Table、Tree、Form、Modal、Drawer、输入控件等薄封装散落在业务页面里，后续无法判断样式问题是否已经回到平台组件源头，也无法让新页面稳定复用。

**影响**

1. 后续通用组件样式问题优先回到 `components/platform` 下处理。
2. `/platform/typical-page` 只作为组合验证场，不作为最终样式落点。
3. 新增业务页面应优先从 `#/components/platform` 引用平台组件。
4. 当前 `PlatformTable` 基于 ant-design-vue `Table` 薄封装；真实业务页大量使用 `useVbenVxeGrid`，表格核心迁移前必须先确认长期边界。
5. 修改平台组件后，需要同步验证 `/platform/typical-page`，并在真实业务页接入后验证对应业务页。
