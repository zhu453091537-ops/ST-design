# 决策记录

本文件记录项目长期有效的关键决策、原因和影响。临时待办写入 `docs/todo-next.md`，阶段性完成记录写入 `docs/project-log.md`。

## 2026-05-05

### 决策 1：Codex 默认作为平台建设战略顾问协作

**决策内容**

在本项目中，Codex 的默认角色不是单纯的代码助手，而是平台建设战略顾问。收到需求后，应优先理解业务目标、分析可行性、整理架构思路、提出技术选型和开发策略；当用户意图不明确时，必须先提问确认。

**原因**

本项目的目标偏向平台建设和长期演进，单点页面实现容易造成规则分散、组件割裂和后续维护成本上升。先分析再实现，可以让每次开发更贴近平台能力建设。

**影响**

1. 不确定需求时，不直接生成代码。
2. 方案明确并得到用户确认后，再进入具体开发建议、代码示例或实际修改。
3. 后续任务需要兼顾业务目标、平台复用性、技术约束和演进成本。

### 决策 2：重要上下文必须沉淀到仓库文档

**决策内容**

项目重要上下文不能只存在聊天记录里。每次开始新任务、接续旧任务、切换电脑、重开聊天，或用户说“开工继续”时，必须先读取：

1. `AGENTS.md`
2. `docs/project-log.md`
3. `docs/decision-records.md`
4. `docs/todo-next.md`

读取后，先输出项目接续摘要，再按用户确认推进。

**原因**

用户会在多台电脑之间切换开发，也可能因为聊天上下文过长而重新开启新聊天。把规则、日志、决策和待办分散在聊天记录里，会导致后续任务无法稳定接续。

**影响**

1. 未读取关键文档前，不允许直接开始大规模修改代码。
2. 阶段完成后必须更新项目日志和下一步待办。
3. 长期规则、技术决策、组件规范、交付规则、禁止事项或平台建设方向，必须同步更新 `docs/decision-records.md` 和 `AGENTS.md`。

### 决策 3：项目文档按职责分工维护

**决策内容**

项目接续文档按职责分工维护：

1. `AGENTS.md`：项目长期规则、开发约束、Codex 执行规范。
2. `docs/project-log.md`：阶段性工作日志，记录已经做过什么。
3. `docs/decision-records.md`：长期有效的关键决策和原因。
4. `docs/todo-next.md`：下一步任务、未完成事项、风险提醒。

**原因**

分开记录长期规则、历史日志、关键决策和当前待办，可以避免文档混杂，方便跨设备和新聊天快速恢复上下文。

**影响**

1. 禁止把临时问题、长期规则、当前待办混在同一个文档里。
2. 后续阶段结束时，需要按内容性质写入对应文档。
3. 用户输入“今天结束”时，必须更新项目日志和下一步待办，并在产生长期规则时同步更新决策记录或 `AGENTS.md`。

### 决策 4：ant-design-vue 组件样式修改默认按平台级改造处理

**决策内容**

本项目基于 Vben Admin 5.0 + ant-design-vue。当用户提出修改 Button、Table、Tree、Form、Input、Select、Modal、Drawer、Tabs、Pagination、SearchForm、列表页、筛选区、弹窗、抽屉等组件样式时，默认理解为基于 ant-design-vue 原生组件进行平台级封装与样式改造，而不是只修改当前业务页面。

**原因**

这些组件通常属于后台管理系统的高频基础能力。如果只在单个页面写局部 CSS，后续会造成样式分散、页面体验不一致、平台组件无法复用、维护成本上升。平台化改造可以统一组件视觉、交互、token、变量和展示页验证。

**影响**

1. 不允许修改 ant-design-vue 源码或 `node_modules`。
2. 不允许引入 Arco、Element Plus、Naive UI 等其他 UI 组件库来替代当前基础组件体系。
3. 不允许只在当前页面写局部 CSS 解决通用组件问题。
4. 平台组件内部可以继续使用 ant-design-vue 原生组件，但业务页面必须优先引用平台组件。
5. 开发或修改前，需要先输出组件映射表，说明页面模块、原生组件、平台组件、是否已有、是否需要改造和样式落点。
6. 如确实只属于当前页面特殊样式，必须先询问用户确认；用户未明确确认“只改当前页面”前，默认按平台级组件改造处理。
7. 修改平台组件后，必须验证典型页面、平台组件展示页、已引用业务页面、直接绕过平台组件的页面，以及页面级样式覆盖风险。
8. 组件改造任务交付报告必须说明修改了哪些平台组件、哪些样式是平台级修改、哪些页面受影响、哪些页面还未接入平台组件、是否存在只改当前页面的特殊情况、当前未完成问题和下一步建议。

### 决策 5：截图与页面需求必须先做结构化分析

**决策内容**

当用户提供截图、截图案例、页面描述，或要求分析列表页、查询页、表单页、弹窗、抽屉、详情页等后台管理页面时，必须先输出页面结构分析、组件映射表、文件归属表、规则自检表、交互与状态说明。分析完成后等待用户确认，再进入页面开发或代码修改。

**原因**

后台管理页面通常由页面标题、查询筛选区、表格列表、操作列、弹窗或抽屉组成。如果没有先把结构、组件来源、文件归属和交互状态说清楚，容易出现页面级临时实现、组件重复建设、样式落点错误或与平台组件治理规则冲突。

**影响**

1. 所有截图和页面需求分析必须使用 Markdown 表格输出组件映射和自检结果。
2. 首选 Vben Admin + ant-design-vue 组件体系，不引入其他 UI 组件库。
3. 文件归属必须基于当前仓库实际结构，如 `apps/web-antd/src/views`、`apps/web-antd/src/components`、`packages/styles` 等，不能直接照搬外部示例路径。
4. 如果截图信息不足以判断业务含义、数据来源、权限规则或交互边界，必须先提问确认。
5. 未经用户确认前，不进入页面开发或代码修改。

### 决策 6：平台组件源头统一放在 components/platform

**决策内容**

当前平台组件源头统一放在 `apps/web-antd/src/components/platform`，并通过 `apps/web-antd/src/components/platform/index.ts` 对外提供出口。典型页面验证场放在 `apps/web-antd/src/views/platform/typical-page/index.vue`，Mock 菜单路径为 `/platform/typical-page`。

**原因**

典型页面驱动平台组件改造需要一个稳定的源头目录和验证入口。如果 Button、Table、Tree、Form、Modal、Drawer、输入控件等薄封装散落在业务页面里，后续无法判断样式问题是否已经回到平台组件源头，也无法让新页面稳定复用。

**影响**

1. 后续通用组件样式问题优先回到 `components/platform` 下处理。
2. `/platform/typical-page` 只作为组合验证场，不作为最终样式落点。
3. 新增业务页面应优先从 `#/components/platform` 引用平台组件。
4. 当前 `PlatformTable` 基于 ant-design-vue `Table` 薄封装；真实业务页大量使用 `useVbenVxeGrid`，表格核心迁移前必须先确认长期边界。
5. 修改平台组件后，需要同步验证 `/platform/typical-page`，并在真实业务页接入后验证对应业务页。

### 决策 7：顶部菜单层级与表格工具栏必须平台源头化

**决策内容**

后台主布局按照典型页面层级组织菜单：顶部导航栏显示一级功能菜单，左侧导航栏显示当前一级功能菜单下的二级、三级菜单。表格常规功能入口设置、刷新、搜索、全屏统一由 `PlatformTableToolbar` 承载。

**原因**

顶部/左侧导航是后台系统框架能力，不属于单个页面。表格右侧常规功能入口也是所有列表页的共性能力，如果在页面内重复手写，会导致按钮默认态、hover、权限扩展和后续交互不一致。

**影响**

1. `header-sidebar-nav` 布局需要拆分菜单数据，顶部只保留一级菜单，左侧显示子菜单。
2. 典型页面和后续列表页需要表格功能区时，应优先使用 `PlatformTableToolbar`。
3. 表格四个常规功能按钮的默认态、hover、focus 等样式从平台组件和全局变量继承。
4. 页面可以保留业务功能按钮数量和权限逻辑，但表格常规工具入口不得在页面内重复手写。
5. 点击有子级的顶部一级菜单时，应默认跳转到该一级菜单下第一个可导航子菜单，使左侧导航同步选中第一个有效菜单项。

### 决策 8：每个步骤完成后必须主动同步方案进度

**决策内容**

每完成一个步骤、一个阶段或一次代码修改后，必须主动输出简短进度报告。报告需要说明当前方案整体进度、已经完成的内容、剩余内容、风险或问题、下一步建议。平台级组件改造还必须说明样式或交互是否已沉淀到源组件、主题变量、全局样式入口或适配层，以及哪些真实业务页还未接入。

**原因**

当前任务属于平台级通用能力建设，涉及 Figma 样式、Ant Design Vue 原生组件、Vben Vxe 表格、导航布局、真实业务页接入等多个阶段。如果只在最终完成时汇报，用户需要反复追问当前进度和剩余范围，不利于跨设备、跨聊天接续。

**影响**

1. 后续每个阶段完成后都要主动汇报，不等用户追问。
2. 汇报不能只写“已完成”或“通过”，必须说明方案进度和剩余内容。
3. 如果没有浏览器验证或只完成文档、审计、局部验证，需要明确说明原因。
4. 该规则适用于所有后续平台级组件、主题、样式、导航、表格、表单、树结构和页面接入任务。

### 决策 9：树筛选容器按平台壳下沉，业务页保留数据与联动逻辑

**决策内容**

对于后台管理页高频出现的“左侧树筛选 + 搜索 + 刷新 + 空态 + 骨架屏”结构，优先沉淀到 `apps/web-antd/src/components/platform/tree/platform-tree-panel.vue` 作为平台壳。真实业务页继续保留树数据请求、筛选联动、查询参数组装、`reload/select` 事件和业务插槽内容。

**原因**

`system/user/dept-tree.vue` 和典型页都存在相同的树筛选结构。如果只在单个页面里组合 `Tree`、`Input`、刷新按钮和空态，会让树结构样式、搜索区间距、按钮状态和后续交互扩展继续分散。把“壳”下沉到平台层，可以统一视觉和交互，同时不强行改动业务页的接口、权限和联动逻辑。

**影响**

1. 树筛选的头部结构、搜索框、刷新按钮、骨架屏和空态优先回到 `PlatformTreePanel` 维护。
2. `PlatformTree` 继续作为树本体样式薄封装，容器级组合能力由 `PlatformTreePanel` 承担。
3. 真实业务页接入树筛选平台化时，应优先复用 `PlatformTreePanel`，而不是在页面内重复手写 `Tree + Input + Button`。
4. 树数据加载、过滤、节点点击联动和查询参数拼装仍保留在业务页面，避免平台组件侵入业务逻辑。

### 决策 10：当前预览只保留平台组件模块

**决策内容**

当前前端预览阶段只保留“平台组件”模块作为唯一可见一级模块，默认首页统一指向 `/platform/typical-page`。Vben 示例的“概览”“关于”“工作台”“更新记录”等模块不再作为当前预览交付范围。

**影响**

2. 前端路由增加平台组件模块，确保混合权限模式下平台组件路由可稳定生成。
3. 旧 `/analytics`、`/workspace`、`/changelog`、`/vben-admin/about` 等地址只保留隐藏重定向到 `/platform/typical-page`，不再显示为可见菜单。
4. 后续如需恢复业务模块，应按真实业务范围重新接入，而不是恢复 Vben 示例模块作为默认导航。

### 决策 11：Figma 连接默认使用本地非官方 Figma MCP Go

**决策内容**

本项目 Figma 连接默认使用用户本机的非官方 `@vkhanhqui/figma-mcp-go`，插件固定连接 `127.0.0.1:1994`，不默认使用官方 Figma MCP。本轮已验证成功的 Codex MCP 配置为直接执行本机缓存的 darwin arm64 二进制：

```bash
codex mcp remove figma-mcp-go
codex mcp add figma-mcp-go -- /Users/zhuguangxiang/.npm/_npx/ae527a30157df656/node_modules/@vkhanhqui/figma-mcp-go/bin/darwin-arm64/figma-mcp-go
codex mcp get figma-mcp-go
```

**原因**

用户明确说明当前使用的不是官方 MCP，且 Figma 插件固定为 `127.0.0.1:1994`。之前通过 `npx -y @vkhanhqui/figma-mcp-go` 间接启动时，出现 `localhost`、`::1` 和服务角色判断相关问题，导致插件显示 `Disconnected`。改为让 Codex MCP 直接执行本机二进制后，插件已显示 `Connected`，并成功读取 Figma 文件、页面和选中节点。

**影响**

1. 后续 Figma 任务先检查 `figma-mcp-go` 本地连接，不要默认切换官方 MCP。
2. Figma 插件必须显示 `127.0.0.1:1994 Connected` 后，再读取节点、截图或设计上下文。
3. 如二进制缓存路径失效，应先在 `~/.npm/_npx` 下重新查找 `@vkhanhqui/figma-mcp-go/bin/darwin-arm64/figma-mcp-go`，再更新 Codex MCP 配置。
4. 若再次出现 `Disconnected`，按 `AGENTS.md` 第 9 节排查。

### 决策 12：典型页面 Demo 必须用真实业务内容验证 ant-design-vue 原生组件体系

**决策内容**

典型页面 Demo 的业务内容必须来自当前项目已有真实业务页面、路由、接口模型或当前 Mock，不得虚构字段、菜单、树节点、表格列和业务动作。组件实现路线必须基于 ant-design-vue 原生组件及其平台薄封装，例如 `a-form`、`a-input`、`a-select`、`a-date-picker`、`a-tree`、`a-table`、`a-pagination`、`a-modal`、`a-drawer`、`a-button`。

**原因**

当前项目需要一个能一眼暴露平台组件问题的真实业务验证场。如果直接复用 `useVbenVxeGrid` 的业务页面形态，无法有效验证 ant-design-vue 原生组件和平台薄封装的后续改造效果；如果虚构业务字段，又会脱离真实管理场景。

**影响**

1. `useVbenVxeGrid` 页面可以作为业务字段、交互和数据来源参考，但不能直接成为 ant-design-vue 典型 Demo 的最终组件形态。
2. 已存在的平台薄封装优先复用，缺失组件先用 ant-design-vue 原生组件或极薄封装承接。
3. 页面级只写布局样式，通用视觉问题后续回到平台组件、主题 token 或全局样式入口。
4. 第一阶段必须先输出真实页面选择、模块清单、组件映射表、开发方案和风险说明，等用户确认后再实现。
5. 后续用户提供原型截图时，必须先将截图元素映射到 ant-design-vue 原生组件和平台薄封装，保证页面之间组件口径一致。
6. 当用户指定修改某个 ant-design-vue 原生组件样式时，默认在本项目全局样式层、主题 token 或平台薄封装源头改造，使同类组件自动全局生效；不得修改 `node_modules` 或 ant-design-vue 源码，也不得在业务页面逐页补丁。

### 决策 13：通用组件样式反馈默认按全局源头处理

**决策内容**

后续用户提供原型截图、Figma 节点、截图案例或页面样式反馈时，只要问题涉及 Button、Table、Tree、Form、Input、Select、DatePicker、Pagination、Modal、Drawer、Tabs、SearchForm、工具栏、操作列、状态标签等通用组件，即使用户没有特别强调“全局修改”，也默认按平台级全局问题处理。

**原因**

用户明确要求避免后续因为没有重复强调而只改当前页面，导致其他页面同类组件没有同步变化。当前项目目标是通过典型页面建立统一组件改造靶场，再把通用样式沉淀到 ant-design-vue 全局样式层、主题 token 或平台薄封装源头，使后续页面自动继承。

**影响**

1. 通用组件样式问题默认优先落到对应平台组件、平台组件内部场景样式、主题变量或 Vben/Antdv 适配层。
2. 未经用户明确确认“只改当前页面”，不得只改页面 scoped CSS。
3. 修改后需要优先验证典型页面，并说明哪些真实业务页会同步受影响。
4. 如果某个问题确实只是当前页面独有布局，应先询问用户确认，不得自行按局部样式处理。
5. 不要轻易直接覆盖 `.ant-*` 全局选择器；只有明确属于全站基础 token、基础控件规范或所有同类 Antdv 组件都应该统一变化时，才允许进入 Antdv 全局样式层。

### 决策 14：典型页面样式沉淀按平台组件边界优先

**决策内容**

典型页面 Demo 的样式沉淀必须优先按照平台组件边界处理，而不是把分页、操作列、状态开关等场景直接写入 Antdv 全局样式。分页优先归入 `PlatformTable` 的 pagination 场景，工具栏归入 `PlatformTableToolbar`，操作列归入 `PlatformTable` 的 action column / action renderer，表格状态切换归入 `PlatformTable` 的 status column 场景。

**原因**

直接覆盖 `.ant-pagination`、`.ant-switch` 等全局选择器风险较高，可能影响全项目所有原生 Antdv 使用场景。当前阶段的目标是用典型页面暴露平台组件边界问题，让真实业务页后续能通过平台组件统一继承，而不是扩大不必要的全局影响面。

**影响**

1. `/platform/typical-page` 的页面级 scoped CSS 只允许写布局，如左右分栏、间距、高度和滚动区域。
2. 分页、工具栏、操作列、状态列、弹窗、抽屉等视觉样式优先回到对应平台组件或组件场景样式。
3. 暂不新增 `PlatformSwitch`；只有后续多个页面在表格外也复用状态开关，或状态开关成为独立平台控件时，再抽成 `PlatformSwitch`。
4. 只有品牌色、字体、基础字号、基础圆角、控件高度、基础边框色等全站基础规范，才允许沉淀到主题 token 或 Antdv 全局样式层。

### 决策 15：典型页面后端不可用时采用专用受控数据源适配文件

**决策内容**

当后端 `8080` 不可用、`/system/user/list` 和 `/system/user/deptTree` 不能直接联调时，`/platform/typical-page` 第一版可以使用页面专用受控数据源适配文件 `apps/web-antd/src/views/platform/typical-page/user-demo-source.ts`。该文件只服务典型页面验证，不污染真实 `/system/user` 页面，不修改全局 mock server，也不把 Mock 数据写进 `index.vue`。

**原因**

用户明确允许在后端暂不可用时补一份“受控 Mock 数据”，但要求字段、columns、接口类型、表单 schema、树结构模型继续来自 `/system/user`，并且未来能无缝切回真实接口。如果把数据写入页面组件或全局 mock server，会让典型页和真实业务页边界变混，也不利于后续恢复真实接口。

**影响**

1. `/platform/typical-page/index.vue` 只依赖 `getTypicalUserList`、`getTypicalDeptTree`、`saveTypicalUser`、`removeTypicalUsers`、`changeTypicalUserStatus` 等数据源方法。
2. 后端恢复后，优先在 `user-demo-source.ts` 内把这些方法切回 `userList`、`getDeptTree`、`userAdd`、`userUpdate`、`userRemove`、`changeUserStatus` 等真实接口，避免重写页面主体。
3. 旧的 `typicalUsers` / `mock.ts` 假数据入口应停止使用并删除。
4. 导入、导出、重置密码等后端不可用入口第一版只保留 UI 和交互入口，不调用不可用接口。

### 决策 16：Vben 核心布局需先做分层归因后再改动

**决策内容**

Vben Layout / Menu / Breadcrumb / Tabs / Route-to-Menu 属于后台框架地基层，不允许为了单个页面效果随意修改。涉及顶部导航、左侧菜单、breadcrumb、tabs / keep-alive、mixed menu、route → menu 映射、权限菜单等能力时，必须先按页面使用方式、路由配置 / menu meta、mock 菜单源、layout / menu / theme 配置、token / CSS 变量、平台组件或项目侧适配层的顺序排查，最后才允许修改 Vben 核心源码。

**原因**

导航和路由映射属于整个平台的公共骨架。如果直接在核心布局层面大改，容易造成当前页面看起来正确，但其他页面、权限路由、标签页和 breadcrumb 同时失真。先分层归因，可以把能在路由、菜单源、token 和项目适配层解决的问题留在低风险层，减少回滚成本。

**影响**

1. 不允许未经确认就重写 Vben Layout 结构。
2. 不允许未经确认就大改 `use-mixed-menu.ts`、route → menu 逻辑、breadcrumb、tabs / keep-alive。
3. 不允许在业务页面手写顶部导航、左侧菜单或 breadcrumb。
4. 通过全局覆盖 `.ant-menu`、`.ant-layout` 粗暴修改导航样式要被视为高风险行为。
5. 如果确实需要修改 Vben 核心布局或菜单逻辑，开发前必须先输出问题归因、影响范围、回滚方案和验证方案。
