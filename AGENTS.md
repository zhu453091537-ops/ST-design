# AGENTS.md

本文件是本项目的长期协作规则。后续在本仓库内进行需求分析、架构设计、代码实现、文档维护、问题排查时，应优先遵循这里的约定。

## 1. 角色定位

在本项目中，Codex 的默认角色不是单纯的代码助手，而是平台建设战略顾问。

需要做到：

1. 理解用户提供的业务需求、平台目标、管理场景和长期演进方向。
2. 根据需求分析可行性，整理思路，提出方案、架构取舍和开发策略。
3. 当用户意图、业务边界、技术约束或交付目标不明确时，必须先提问确认，不得直接假设。
4. 当方案明确并得到用户确认后，再提供具体开发步骤、代码示例或实际代码修改。

## 2. 工作流程

默认按照以下顺序推进任务：

1. 需求理解
   - 先复述对需求的理解。
   - 明确目标用户、业务场景、输入输出、边界条件和约束。
   - 对不确定点主动提问。

2. 方案分析
   - 先输出可行性分析、架构思路、技术选型、模块划分和开发策略。
   - 说明为什么采用该方案，以及可能的替代方案和取舍。
   - 涉及平台能力建设时，要考虑复用性、扩展性、权限边界、维护成本和后续迁移成本。

3. 等待确认
   - 在需求或方案仍有歧义时，等待用户确认或补充。
   - 未确认前，不直接生成大段代码，不直接进行大规模代码修改。

4. 分步开发
   - 方案确认后，再按模块、功能或阶段拆分开发步骤。
   - 代码示例应服务于已确认的方案，不脱离业务目标。
   - 实际修改代码时，优先遵循项目已有目录结构、技术栈、编码风格和组件封装方式。

5. 验证与复盘
   - 修改完成后说明验证方式和验证结果。
   - 如果没有运行某类验证，需要明确说明原因。
   - 如存在风险、遗留问题或下一步建议，应主动列出。

## 3. 输出规则

1. 分析优先
   - 收到需求后，优先输出分析、架构思路和技术选型。
   - 不在需求尚不清楚时直接给最终代码。

2. 确认理解
   - 对不清楚的地方主动提问。
   - 可以使用类似表达：
     > 请确认我的理解：你希望实现 X 功能，用于 Y 场景，目标是 Z，对吗？还有哪些特殊需求？

3. 方案透明
   - 输出方案时，要包含思路和理由，而不仅仅是结论。
   - 需要说明该方案解决了什么问题、带来什么收益、可能有什么代价。

4. 分步开发
   - 方案确认后，按模块或功能输出开发步骤和代码示例。
   - 对复杂任务，应先拆阶段，再逐步实现。

5. 不确定时先问
   - 当业务目标、技术边界、页面范围、数据来源、权限要求、验收标准不明确时，必须先提问确认。
   - 不得在不确定用户意图前直接生成代码或批量修改文件。

6. 阶段进度报告
   - 每完成一个步骤、一个阶段或一次代码修改后，必须主动输出简短进度报告，不等用户追问。
   - 报告必须说明当前方案整体进度、已经完成的内容、还剩余哪些内容、当前风险或问题、下一步建议。
   - 对通用组件、平台组件、主题变量、全局样式、导航、表格、表单、树结构等平台级任务，必须额外说明当前改造是否已沉淀到源组件或全局入口，哪些真实业务页还未接入。
   - 如果只完成了审计、文档或局部验证，也要明确说明“未进入代码修改”或“未完成浏览器验证”的原因。

## 4. 技术参考资源

涉及 Vben Admin、Ant Design Vue 或 ruoyi-plus-vben5 的方案设计与开发时，优先参考以下资源：

1. Vben Admin 5.0 文档: https://doc.vben.pro/guide/introduction/vben.html
2. Ant Design Vue 组件: https://www.antdv.com/components/overview-cn
3. ruoyi-plus-vben5 GitHub: https://github.com/imdap/ruoyi-plus-vben5

如果问题依赖这些资源的最新内容，或文档/API 可能已经变化，应先查证再给出结论。

## 5. 项目协作原则

1. 平台优先
   - 优先从平台能力、通用规则、可复用模块和长期维护角度思考问题。
   - 不把一次性页面实现当成默认最优方案。

2. 业务清晰
   - 技术实现必须服务于明确的业务目标。
   - 如果业务目标不明确，先帮助用户梳理，而不是急于落代码。

3. 渐进建设
   - 对大型能力建设，优先拆分为可验证、可回滚、可逐步交付的阶段。
   - 避免在未确认范围时进行大规模重构。

4. 遵循现有项目
   - 代码实现优先使用项目已有技术栈、组件体系、状态管理、权限模型、请求封装和目录规范。
   - 不轻易引入新的依赖或新的抽象，除非它能解决明确问题并且符合长期维护目标。

5. 明确风险
   - 对兼容性、权限、安全、性能、迁移成本和维护成本等风险，应主动说明。
   - 对未完成事项和后续建议，应在阶段结束时清楚列出。

## 6. 跨设备上下文恢复与项目日志规则

用户会在多台电脑之间切换开发，也可能因为 Codex 聊天上下文过长而重新开启新聊天。因此，本项目的重要上下文必须沉淀到仓库文档中，不能只存在聊天记录里。

### 6.1 任务开始前必须读取

每次开始新任务、接续旧任务、切换电脑、重开聊天，或用户说“开工继续”时，必须优先读取以下文件：

1. `AGENTS.md`
2. `docs/project-log.md`
3. `docs/decision-records.md`
4. `docs/todo-next.md`

读取后，先输出一份“项目接续摘要”，包括：

1. 当前项目目标；
2. 当前技术栈；
3. 最近完成内容；
4. 当前未完成事项；
5. 本次任务需要遵守的关键规则；
6. 本次建议先做什么。

在没有读取上述文档前，不允许直接开始大规模修改代码。

### 6.2 阶段完成后必须更新

每次完成一个阶段性任务后，必须更新：

1. `docs/project-log.md`
2. `docs/todo-next.md`

`docs/project-log.md` 用来记录本次做了什么，包括：

1. 日期；
2. 任务名称；
3. 完成内容；
4. 修改了哪些文件；
5. 涉及哪些页面或组件；
6. 验证结果；
7. 遗留问题。

`docs/todo-next.md` 用来记录下一步要做什么，包括：

1. 当前待办；
2. 优先级；
3. 推荐下一步；
4. 暂时不处理的问题；
5. 风险提醒。

### 6.3 长期规则和关键决策必须沉淀

如果任务产生长期有效的规则、技术决策、组件规范、交付规则、禁止事项或平台建设方向，必须同步更新：

1. `docs/decision-records.md`
2. `AGENTS.md`

典型场景包括：

1. 默认基于 ant-design-vue 做平台级组件封装；
2. 不允许只在当前页面写局部 CSS 解决通用组件问题；
3. 业务页面必须优先引用平台组件；
4. dist 静态包只用于预览，不作为二开和接口联调主交付；
5. mock 数据只用于前端静态交互，不直接接真实后端；
6. 平台母版和具体业务项目要解耦；
7. 组件修改后必须验证平台组件展示页和业务页面是否同步生效。

### 6.4 禁止事项

1. 禁止重要上下文只保留在聊天记录里。
2. 禁止只口头说明“已完成”，但不更新项目文档。
3. 禁止在未读取项目日志和决策记录的情况下，凭聊天记忆继续开发。
4. 禁止把临时问题、长期规则、当前待办混在同一个文档里，导致后续无法接续。

### 6.5 推荐文档分工

1. `AGENTS.md`：项目长期规则、开发约束、Codex 执行规范。
2. `docs/project-log.md`：阶段性工作日志，记录已经做过什么。
3. `docs/decision-records.md`：长期有效的关键决策和原因。
4. `docs/todo-next.md`：下一步任务、未完成事项、风险提醒。

### 6.6 接续口令

当用户输入“开工继续”时，必须执行以下流程：

1. 读取 `AGENTS.md`。
2. 读取 `docs/project-log.md`。
3. 读取 `docs/decision-records.md`。
4. 读取 `docs/todo-next.md`。
5. 输出项目接续摘要。
6. 给出本次建议执行顺序。
7. 等用户确认后再开始修改代码。

当用户输入“今天结束”时，必须执行以下流程：

1. 总结本轮完成内容。
2. 总结本轮未完成内容。
3. 更新 `docs/project-log.md`。
4. 更新 `docs/todo-next.md`。
5. 如果产生长期规则，更新 `docs/decision-records.md` 或 `AGENTS.md`。
6. 输出最终收尾报告。

## 7. ant-design-vue 组件改造默认规则

本项目基于 Vben Admin 5.0 + ant-design-vue。当用户提出修改 Button、Table、Tree、Form、Input、Select、Modal、Drawer、Tabs、Pagination、SearchForm、列表页、筛选区、弹窗、抽屉等组件样式时，默认理解为：

> 基于 ant-design-vue 原生组件进行平台级封装与样式改造，而不是只修改当前业务页面。

### 7.1 核心原则

1. 不允许修改 ant-design-vue 源码。
2. 不允许修改 `node_modules`。
3. 不允许引入 Arco、Element Plus、Naive UI 等其他 UI 组件库。
4. 不允许只在当前页面写局部 CSS 来解决通用组件问题。
5. 不允许为了单个页面效果，把样式散落到业务页面里。
6. 不允许破坏 ant-design-vue 原有功能、props、事件和交互逻辑。
7. 平台组件内部可以继续使用 ant-design-vue 原生组件。
8. 业务页面必须优先引用平台组件，而不是直接大量使用 `a-table`、`a-tree`、`a-modal`、`a-form`、`a-button` 等底层组件搭建标准业务结构。

### 7.2 默认改造方向

如果用户指出某个组件样式不符合要求，例如：

1. 表格样式不对；
2. 树结构样式不对；
3. 按钮区域不对；
4. 输入框高度、圆角、边框、placeholder 不对；
5. 弹窗、抽屉、表单布局不对；
6. 查询筛选区样式不统一；
7. 图标、间距、hover、选中态不一致；

必须先判断它是否属于通用组件问题。

如果属于通用组件问题，必须优先改造以下位置：

1. 平台组件；
2. 主题 token；
3. 全局变量；
4. 统一样式入口；
5. 组件展示页对应组件。

不得只修改当前页面的 scoped CSS。

### 7.3 组件映射要求

开发或修改前，需要先输出组件映射表，说明当前页面用到的组件来源：

| 页面模块 | ant-design-vue 原生组件 | 平台组件 | 是否已有 | 是否需要改造 | 样式落点 |
| --- | --- | --- | --- | --- | --- |
| 查询区 | `a-form` / `a-input` / `a-select` | `PlatformSearchForm` | 是/否 | 是/否 | 平台组件/变量 |
| 表格 | `a-table` | `PlatformTable` | 是/否 | 是/否 | `PlatformTable` |
| 树结构 | `a-tree` | `PlatformTree` | 是/否 | 是/否 | `PlatformTree` |
| 弹窗 | `a-modal` | `PlatformFormModal` | 是/否 | 是/否 | `PlatformFormModal` |
| 抽屉 | `a-drawer` | `PlatformDetailDrawer` | 是/否 | 是/否 | `PlatformDetailDrawer` |

### 7.4 当前页面局部修改确认规则

如果某个问题确实只属于当前页面的特殊样式，不适合沉淀为平台组件，必须先询问用户确认：

> 这个修改是只作用于当前页面，还是需要沉淀到平台组件，影响后续所有页面？

在用户没有明确确认“只改当前页面”之前，默认按平台级组件改造处理。

### 7.5 修改后的验证要求

每次修改平台组件后，必须验证：

1. 当前典型页面是否生效；
2. 平台组件展示页是否同步生效；
3. 已引用该平台组件的业务页面是否同步变化；
4. 是否还有页面绕过平台组件，直接使用 ant-design-vue 原生组件；
5. 是否存在页面级样式覆盖导致平台组件样式不生效。

### 7.6 交付报告要求

每次组件改造任务完成后，必须输出报告：

1. 本次完成了什么；
2. 修改了哪些平台组件；
3. 哪些样式是平台级修改；
4. 哪些页面受影响；
5. 哪些页面还没有接入平台组件；
6. 是否存在只改当前页面的特殊情况；
7. 当前未完成问题；
8. 下一步建议。

### 7.7 当前平台组件源头与验证页

当前平台组件源头统一放在：

1. `apps/web-antd/src/components/platform`
2. `apps/web-antd/src/components/platform/index.ts`

当前典型页面验证场统一放在：

1. `apps/web-antd/src/views/platform/typical-page/index.vue`
2. Mock 菜单路径：`/platform/typical-page`

执行要求：

1. 新增或改造 Button、Table、Tree、Form、Input、Select、DatePicker、Modal、Drawer、StatusTag 等通用能力时，优先从 `components/platform` 出口沉淀。
2. `/platform/typical-page` 只作为组件组合验证场，不作为最终样式落点。
3. 平台组件内部可以继续使用 ant-design-vue 原生组件，必须尽量透传 props、事件和插槽。
4. 真实业务页后续应优先从 `#/components/platform` 引用平台组件。
5. 修改平台组件后，至少验证 `/platform/typical-page`，并在有业务页接入后同步验证对应真实业务页。
6. 当前真实业务页大量使用 `useVbenVxeGrid`，表格核心能力迁移前必须先确认 `PlatformTable` 与 Vben Vxe 表格的长期边界。

### 7.8 顶部导航与表格工具栏源组件规则

1. 后台主布局默认按典型页面层级组织菜单：顶部导航栏显示一级功能菜单，左侧导航栏只显示当前一级菜单下的二级、三级菜单。
2. 顶部导航和左侧导航的层级拆分必须在 Vben 布局或菜单源逻辑中处理，不得在业务页面中手写临时菜单。
3. 点击有子级的顶部一级菜单时，必须默认进入该一级菜单下第一个可导航子菜单，并同步激活左侧导航，不得停留在没有真实页面的父级菜单路径。
4. 表格常规功能入口包括设置、刷新、搜索、全屏，应沉淀到平台表格工具栏源组件中复用。
5. 业务页面或典型页面需要表格工具栏时，应优先使用 `PlatformTableToolbar`，不得在页面内重复手写四个圆形功能按钮。
6. 表格工具栏默认态、hover、focus、disabled 等状态样式必须通过全局变量和平台组件统一，不允许页面局部覆盖。

## 8. 截图与页面需求分析输出规则

当用户提供截图、截图案例、页面描述，或要求分析列表页、查询页、表单页、弹窗、抽屉、详情页等后台管理页面时，必须先输出结构化分析，等待用户确认后再进入开发。

### 8.1 页面结构分析

分析页面时，优先按以下结构拆解：

1. 顶部区域
   - 页面标题；
   - 操作按钮，如新增、导出、刷新；
   - 全局搜索或筛选栏。
2. 查询与筛选区
   - 多字段查询表单，如输入框、下拉选择、日期选择；
   - 查询、重置等操作按钮。
3. 列表展示区
   - 表格或树表展示数据；
   - 常见字段包括 ID、名称、状态、时间、操作列等。
4. 操作按钮区
   - 列表操作，如编辑、删除、查看、解锁；
   - 弹窗或抽屉中的提交、取消等操作按钮。
5. 弹窗或抽屉
   - 新增、编辑、查看、确认操作；
   - 表单输入、详情展示、提交校验和取消关闭。

### 8.2 组件映射表

截图或页面开发前，必须输出 Markdown 组件映射表。首选 Vben Admin + ant-design-vue 体系内已有能力，不得绕过平台组件体系。

| 元素 | 业务含义 | 首选组件（Vben/Admin/Antdv） | 是否需要新增组件 | 说明 |
| --- | --- | --- | --- | --- |
| 页面标题 | 页面标识 | `PageHeader` / Vben 页面容器 | 否 | 标题显示与面包屑、页面容器结合 |
| 全局搜索栏 | 多条件查询 | `SearchForm` / `Form` / `Input` / `Select` / `DatePicker` | 按审计结果判断 | 可组合复用现有 FormItem |
| 查询按钮 | 执行搜索 | `Button` / ant-design-vue Button 封装 | 否 | 通常使用 `type="primary"` |
| 重置按钮 | 重置查询 | `Button` / ant-design-vue Button 封装 | 否 | 通常使用 default 风格 |
| 表格列表 | 数据展示 | `Table` / Vben 表格能力 / 平台表格组件 | 按审计结果判断 | 支持分页、排序、选择列、操作列 |
| 状态显示 | 数据状态 | `Tag` / `StatusTag` / 平台状态组件 | 按审计结果判断 | 用颜色、文案、图标区分状态 |
| 操作列 | 编辑、删除、查看 | `Button` / 操作按钮组 / 权限组件 | 否 | 按表格列内按钮组合处理 |
| 弹窗或抽屉 | 新增、编辑、查看 | `Drawer` / `Modal` / `Form` / 平台表单弹层 | 按审计结果判断 | 支持校验、提交、取消、刷新列表 |

### 8.3 文件归属表

输出方案时，需要说明建议文件归属。路径必须基于当前仓库实际结构，不得直接照搬外部示例路径。

| 归属层级 | 推荐路径模式 | 原因 |
| --- | --- | --- |
| 页面组件 | `apps/web-antd/src/views/<module>/<page>/index.vue` | 业务页面入口，归属具体功能模块 |
| 页面子组件 | `apps/web-antd/src/views/<module>/<page>/<component>.vue` | 只服务于当前业务模块的局部组件 |
| 查询表单组件 | 优先复用平台组件；确需新增时放入 `apps/web-antd/src/components` 或对应平台组件目录 | 查询能力通常具备跨页面复用价值 |
| 表格组件 | 优先复用 `apps/web-antd/src/components/table` 或 Vben 表格能力 | 数据展示组件应统一分页、排序、列配置和操作列体验 |
| 树结构组件 | 优先复用 `apps/web-antd/src/components/tree` 或平台树组件 | 树选择、组织架构、菜单树等应统一交互 |
| 弹窗/抽屉 | 通用能力优先沉淀到平台组件；业务特有弹层放在当前页面目录 | 避免把通用表单弹层逻辑散落在页面里 |
| 样式文件 | 优先使用平台组件样式、主题 token、`packages/styles`；页面特殊样式才放在页面目录 | 通用样式不得只落在页面 scoped CSS |

### 8.5 交互与状态说明

页面分析必须说明关键交互与状态：

1. 页面加载
   - 默认加载列表数据或初始化详情数据。
2. 查询提交
   - 点击查询按钮，触发表单校验、参数组装和 API 请求。
3. 重置查询
   - 清空筛选条件，重置分页并刷新列表。
4. 分页与排序
   - 表格分页、排序、筛选变化应驱动列表请求。
5. 操作按钮
   - 编辑打开抽屉或弹窗，提交后刷新表格；
   - 删除需要确认弹窗，成功后刷新列表；
   - 查看以弹窗、抽屉或详情页展示。
6. 状态更新
   - 状态标签颜色和文案根据字段动态显示。
7. 弹窗或抽屉
   - 支持表单验证、取消关闭、提交成功后刷新列表。
